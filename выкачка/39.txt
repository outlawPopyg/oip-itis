<p><b>Docker</b> is a set of <a href="/wiki/Platform_as_a_service" title="Platform as a service">platform as a service</a> (PaaS) products that use <a href="/wiki/OS-level_virtualization" title="OS-level virtualization">OS-level virtualization</a> to deliver software in packages called <i><a href="/wiki/Container_(virtualization)" class="mw-redirect" title="Container (virtualization)">containers</a></i>.<sup id="cite_ref-SYS-CON_Media_5-0" class="reference"><a href="#cite_note-SYS-CON_Media-5"><span class="cite-bracket">&#91;</span>5<span class="cite-bracket">&#93;</span></a></sup> 
The service has both free and premium tiers. The software that hosts the containers is called <b>Docker Engine</b>.<sup id="cite_ref-what-is-a-container_6-0" class="reference"><a href="#cite_note-what-is-a-container-6"><span class="cite-bracket">&#91;</span>6<span class="cite-bracket">&#93;</span></a></sup> It was first released in 2013 and is developed by <a href="/wiki/Docker,_Inc." title="Docker, Inc.">Docker, Inc.</a><sup id="cite_ref-os4u_7-0" class="reference"><a href="#cite_note-os4u-7"><span class="cite-bracket">&#91;</span>7<span class="cite-bracket">&#93;</span></a></sup>
</p><p>Docker is a tool that is used to automate the deployment of <a href="/wiki/Application_software" title="Application software">applications</a> in lightweight containers so that applications can work efficiently in different environments in isolation.
</p>
<meta property="mw:PageProp/toc" />
<div class="mw-heading mw-heading2"><h2 id="Background">Background</h2><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Docker_(software)&amp;action=edit&amp;section=1" title="Edit section: Background"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></div>
<p>Containers are isolated from one another and bundle their own software, <a href="/wiki/Library_(computing)" title="Library (computing)">libraries</a> and configuration files; they can communicate with each other through well-defined channels.<sup id="cite_ref-Auto38-1_8-0" class="reference"><a href="#cite_note-Auto38-1-8"><span class="cite-bracket">&#91;</span>8<span class="cite-bracket">&#93;</span></a></sup> Because all of the containers share the services of a single <a href="/wiki/Kernel_(operating_system)" title="Kernel (operating system)">operating system kernel</a>, they use fewer resources than <a href="/wiki/Virtual_machine" title="Virtual machine">virtual machines</a>.<sup id="cite_ref-what-is-a-container_6-1" class="reference"><a href="#cite_note-what-is-a-container-6"><span class="cite-bracket">&#91;</span>6<span class="cite-bracket">&#93;</span></a></sup>
</p>
<div class="mw-heading mw-heading3"><h3 id="Operation">Operation</h3><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Docker_(software)&amp;action=edit&amp;section=2" title="Edit section: Operation"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></div>
<figure class="mw-default-size" typeof="mw:File/Thumb"><a href="/wiki/File:Docker-linux-interfaces.svg" class="mw-file-description"><img src="//upload.wikimedia.org/wikipedia/commons/thumb/0/09/Docker-linux-interfaces.svg/220px-Docker-linux-interfaces.svg.png" decoding="async" width="220" height="203" class="mw-file-element" srcset="//upload.wikimedia.org/wikipedia/commons/thumb/0/09/Docker-linux-interfaces.svg/330px-Docker-linux-interfaces.svg.png 1.5x, //upload.wikimedia.org/wikipedia/commons/thumb/0/09/Docker-linux-interfaces.svg/440px-Docker-linux-interfaces.svg.png 2x" data-file-width="720" data-file-height="664" /></a><figcaption>Docker can use different interfaces to access virtualization features of the Linux kernel.<sup id="cite_ref-docker-blog-2014-03_9-0" class="reference"><a href="#cite_note-docker-blog-2014-03-9"><span class="cite-bracket">&#91;</span>9<span class="cite-bracket">&#93;</span></a></sup></figcaption></figure>
<p>Docker can package an application and its dependencies in a virtual container that can run on any <a href="/wiki/Linux" title="Linux">Linux</a>, <a href="/wiki/Microsoft_Windows" title="Microsoft Windows">Windows</a>, or <a href="/wiki/MacOS" title="MacOS">macOS</a> computer. This enables the application to run in a variety of locations, such as <a href="/wiki/On-premises_software" title="On-premises software">on-premises</a>, in <a href="/wiki/Public_cloud" class="mw-redirect" title="Public cloud">public</a> <i>(see <a href="/wiki/Decentralized_computing" title="Decentralized computing">decentralized computing</a>, <a href="/wiki/Distributed_computing" title="Distributed computing">distributed computing</a>, and <a href="/wiki/Cloud_computing" title="Cloud computing">cloud computing</a>)</i> or <a href="/wiki/Private_cloud" class="mw-redirect" title="Private cloud">private cloud</a>.<sup id="cite_ref-Linux_10-0" class="reference"><a href="#cite_note-Linux-10"><span class="cite-bracket">&#91;</span>10<span class="cite-bracket">&#93;</span></a></sup> When running on Linux, Docker uses the resource isolation features of the <a href="/wiki/Linux_kernel" title="Linux kernel">Linux kernel</a> (such as <a href="/wiki/Cgroups" title="Cgroups">cgroups</a> and kernel <a href="/wiki/Linux_namespaces" title="Linux namespaces">namespaces</a>) and a <a href="/wiki/Union_mount" title="Union mount">union-capable file system</a> (such as <a href="/wiki/OverlayFS" title="OverlayFS">OverlayFS</a>)<sup id="cite_ref-select-storage-driver_11-0" class="reference"><a href="#cite_note-select-storage-driver-11"><span class="cite-bracket">&#91;</span>11<span class="cite-bracket">&#93;</span></a></sup> to allow containers to run within a single Linux instance, avoiding the overhead of starting and maintaining <a href="/wiki/Virtual_machine" title="Virtual machine">virtual machines</a>. Docker on <a href="/wiki/MacOS" title="MacOS">macOS</a> uses a Linux <a href="/wiki/Virtual_machine" title="Virtual machine">virtual machine</a> to run the containers.<sup id="cite_ref-docker-for-mac_12-0" class="reference"><a href="#cite_note-docker-for-mac-12"><span class="cite-bracket">&#91;</span>12<span class="cite-bracket">&#93;</span></a></sup>
</p><p>Because Docker containers are <a href="/wiki/Light-weight_Linux_distribution" title="Light-weight Linux distribution">lightweight</a>, a single server or virtual machine can run several containers simultaneously.<sup id="cite_ref-Chris_K.,_Poweruser_2019_13-0" class="reference"><a href="#cite_note-Chris_K.,_Poweruser_2019-13"><span class="cite-bracket">&#91;</span>13<span class="cite-bracket">&#93;</span></a></sup> A 2018 analysis found that a typical Docker use case involves running eight containers per host, and that a quarter of analyzed organizations run 18 or more per host.<sup id="cite_ref-datadog_14-0" class="reference"><a href="#cite_note-datadog-14"><span class="cite-bracket">&#91;</span>14<span class="cite-bracket">&#93;</span></a></sup> It can also be installed on a single board computer like the <a href="/wiki/Raspberry_Pi" title="Raspberry Pi">Raspberry Pi</a>.<sup id="cite_ref-15" class="reference"><a href="#cite_note-15"><span class="cite-bracket">&#91;</span>15<span class="cite-bracket">&#93;</span></a></sup>
</p><p><a href="/wiki/Linux_kernel" title="Linux kernel">The Linux kernel's</a> support for namespaces mostly<sup id="cite_ref-kernel-keyrings_16-0" class="reference"><a href="#cite_note-kernel-keyrings-16"><span class="cite-bracket">&#91;</span>16<span class="cite-bracket">&#93;</span></a></sup> isolates an application's view of the operating environment, including process trees, network, user IDs and mounted file systems, while the kernel's cgroups provide resource limiting for memory and CPU.<sup id="cite_ref-limit-resources_17-0" class="reference"><a href="#cite_note-limit-resources-17"><span class="cite-bracket">&#91;</span>17<span class="cite-bracket">&#93;</span></a></sup> Since version 0.9, Docker includes its own component (called <style data-mw-deduplicate="TemplateStyles:r886049734">.mw-parser-output .monospaced{font-family:monospace,monospace}</style><span class="monospaced">libcontainer</span>) to use virtualization facilities provided directly by the Linux kernel, in addition to using abstracted virtualization interfaces via <a href="/wiki/Libvirt" title="Libvirt">libvirt</a>, <a href="/wiki/LXC" title="LXC">LXC</a> and <a href="/wiki/Systemd-nspawn" class="mw-redirect" title="Systemd-nspawn">systemd-nspawn</a>.<sup id="cite_ref-zdnet-7000030397_18-0" class="reference"><a href="#cite_note-zdnet-7000030397-18"><span class="cite-bracket">&#91;</span>18<span class="cite-bracket">&#93;</span></a></sup><sup id="cite_ref-docker-blog-2014-03_9-1" class="reference"><a href="#cite_note-docker-blog-2014-03-9"><span class="cite-bracket">&#91;</span>9<span class="cite-bracket">&#93;</span></a></sup><sup id="cite_ref-Linux_10-1" class="reference"><a href="#cite_note-Linux-10"><span class="cite-bracket">&#91;</span>10<span class="cite-bracket">&#93;</span></a></sup><sup id="cite_ref-libcontainer_19-0" class="reference"><a href="#cite_note-libcontainer-19"><span class="cite-bracket">&#91;</span>19<span class="cite-bracket">&#93;</span></a></sup>
</p><p>Docker implements a high-level <a href="/wiki/API" title="API">API</a> to provide lightweight containers that run processes in isolation.<sup id="cite_ref-infoq-2013-03_20-0" class="reference"><a href="#cite_note-infoq-2013-03-20"><span class="cite-bracket">&#91;</span>20<span class="cite-bracket">&#93;</span></a></sup>
</p>
<div class="mw-heading mw-heading2"><h2 id="Licensing_model">Licensing model</h2><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Docker_(software)&amp;action=edit&amp;section=3" title="Edit section: Licensing model"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></div>
<ul><li>The Docker Engine is licensed under the <a href="/wiki/Apache_License_2.0" class="mw-redirect" title="Apache License 2.0">Apache License 2.0</a>. Docker Desktop distributes some components that are licensed under the <a href="/wiki/GNU_General_Public_License" title="GNU General Public License">GNU General Public License</a>. Docker Desktop is not free for large enterprises.<sup id="cite_ref-21" class="reference"><a href="#cite_note-21"><span class="cite-bracket">&#91;</span>21<span class="cite-bracket">&#93;</span></a></sup></li>
<li>The Dockerfile files can be licensed under an open-source license themselves. The scope of such a license statement is only the Dockerfile and not the container image.</li></ul>
<div class="mw-heading mw-heading2"><h2 id="Components">Components</h2><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Docker_(software)&amp;action=edit&amp;section=4" title="Edit section: Components"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></div>
<p>The Docker software as a service offering consists of three components:
</p>
<ul><li><b>Software:</b> The Docker <a href="/wiki/Daemon_(computing)" title="Daemon (computing)"><b>daemon</b></a>, called <code>dockerd</code>, is a persistent process that manages Docker containers and handles container objects. The daemon listens for requests sent via the Docker Engine API.<sup id="cite_ref-docker-overview_22-0" class="reference"><a href="#cite_note-docker-overview-22"><span class="cite-bracket">&#91;</span>22<span class="cite-bracket">&#93;</span></a></sup><sup id="cite_ref-dockerd_23-0" class="reference"><a href="#cite_note-dockerd-23"><span class="cite-bracket">&#91;</span>23<span class="cite-bracket">&#93;</span></a></sup> The Docker client program, called <code>docker</code>, provides a <a href="/wiki/Command-line_interface" title="Command-line interface">command-line interface</a> (CLI) that allows users to interact with Docker daemons.<sup id="cite_ref-docker-overview_22-1" class="reference"><a href="#cite_note-docker-overview-22"><span class="cite-bracket">&#91;</span>22<span class="cite-bracket">&#93;</span></a></sup><sup id="cite_ref-cli-usage_24-0" class="reference"><a href="#cite_note-cli-usage-24"><span class="cite-bracket">&#91;</span>24<span class="cite-bracket">&#93;</span></a></sup></li>
<li><b>Objects:</b> Docker objects are various entities used to assemble an application in Docker. The main classes of Docker objects are images, containers, and services.<sup id="cite_ref-docker-overview_22-2" class="reference"><a href="#cite_note-docker-overview-22"><span class="cite-bracket">&#91;</span>22<span class="cite-bracket">&#93;</span></a></sup>
<ul><li>A Docker <b>container</b> is a standardized, encapsulated environment that runs applications.<sup id="cite_ref-digitalocean_25-0" class="reference"><a href="#cite_note-digitalocean-25"><span class="cite-bracket">&#91;</span>25<span class="cite-bracket">&#93;</span></a></sup> A container is managed using the Docker API or <a href="/wiki/Command-line_interface" title="Command-line interface">CLI</a>.<sup id="cite_ref-docker-overview_22-3" class="reference"><a href="#cite_note-docker-overview-22"><span class="cite-bracket">&#91;</span>22<span class="cite-bracket">&#93;</span></a></sup></li>
<li>A Docker <b>image</b> is a read-only template used to build containers. Images are used to store and ship applications.<sup id="cite_ref-docker-overview_22-4" class="reference"><a href="#cite_note-docker-overview-22"><span class="cite-bracket">&#91;</span>22<span class="cite-bracket">&#93;</span></a></sup></li>
<li>A Docker <b>service</b> allows containers to be scaled across multiple Docker daemons. The result is known as a <i>swarm</i>, a set of cooperating daemons that communicate through the Docker API.<sup id="cite_ref-docker-overview_22-5" class="reference"><a href="#cite_note-docker-overview-22"><span class="cite-bracket">&#91;</span>22<span class="cite-bracket">&#93;</span></a></sup></li></ul></li>
<li><b>Registries:</b> A Docker registry is a repository for Docker images. Docker clients connect to registries to download ("pull") images for use or upload ("push") images that they have built. Registries can be public or private. The main public registry is Docker Hub. Docker Hub is the default registry where Docker looks for images.<sup id="cite_ref-docker-overview_22-6" class="reference"><a href="#cite_note-docker-overview-22"><span class="cite-bracket">&#91;</span>22<span class="cite-bracket">&#93;</span></a></sup><sup id="cite_ref-docker-registry_26-0" class="reference"><a href="#cite_note-docker-registry-26"><span class="cite-bracket">&#91;</span>26<span class="cite-bracket">&#93;</span></a></sup> Docker registries also allow the creation of notifications based on events.<sup id="cite_ref-Auto38-2_27-0" class="reference"><a href="#cite_note-Auto38-2-27"><span class="cite-bracket">&#91;</span>27<span class="cite-bracket">&#93;</span></a></sup></li></ul>

<div class="mw-heading mw-heading3"><h3 id="Tools">Tools</h3><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Docker_(software)&amp;action=edit&amp;section=6" title="Edit section: Tools"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></div>
<ul><li><b>Docker Compose</b> is a tool for defining and running multi-container Docker applications.<sup id="cite_ref-compose-overview_29-0" class="reference"><a href="#cite_note-compose-overview-29"><span class="cite-bracket">&#91;</span>29<span class="cite-bracket">&#93;</span></a></sup> It uses <a href="/wiki/YAML" title="YAML">YAML</a> files to configure the application's services and performs the creation and start-up process of all the containers with a single command. The <code>docker compose</code> CLI utility allows users to run commands on multiple containers at once; for example, building images, <a href="/wiki/Scalability" title="Scalability">scaling</a> containers, running containers that were stopped, and more.<sup id="cite_ref-compose-cli-ref_30-0" class="reference"><a href="#cite_note-compose-cli-ref-30"><span class="cite-bracket">&#91;</span>30<span class="cite-bracket">&#93;</span></a></sup> Commands related to image manipulation, or user-interactive options, are not relevant in Docker Compose because they address one container.<sup id="cite_ref-codeship_31-0" class="reference"><a href="#cite_note-codeship-31"><span class="cite-bracket">&#91;</span>31<span class="cite-bracket">&#93;</span></a></sup> The <b>docker-compose.yml</b> file is used to define an application's services and includes various configuration options. For example, the <code>build</code> option defines configuration options such as the Dockerfile path, the <code>command</code> option allows one to override default Docker commands, and more.<sup id="cite_ref-compose-v3-ref_32-0" class="reference"><a href="#cite_note-compose-v3-ref-32"><span class="cite-bracket">&#91;</span>32<span class="cite-bracket">&#93;</span></a></sup> The first public beta version of Docker Compose (version 0.0.1) was released on December 21, 2013.<sup id="cite_ref-release-0.0.1_33-0" class="reference"><a href="#cite_note-release-0.0.1-33"><span class="cite-bracket">&#91;</span>33<span class="cite-bracket">&#93;</span></a></sup> The first production-ready version (1.0) was made available on October 16, 2014.<sup id="cite_ref-release-1.0.0_34-0" class="reference"><a href="#cite_note-release-1.0.0-34"><span class="cite-bracket">&#91;</span>34<span class="cite-bracket">&#93;</span></a></sup></li>
<li><b>Docker Swarm</b> provides native <a href="/wiki/Computer_cluster" title="Computer cluster">clustering</a> functionality for Docker containers, which turns a group of Docker engines into a single virtual Docker engine.<sup id="cite_ref-container-orchestration-tools_35-0" class="reference"><a href="#cite_note-container-orchestration-tools-35"><span class="cite-bracket">&#91;</span>35<span class="cite-bracket">&#93;</span></a></sup> In Docker 1.12 and higher, Swarm mode is integrated with Docker Engine.<sup id="cite_ref-docker-swarm_36-0" class="reference"><a href="#cite_note-docker-swarm-36"><span class="cite-bracket">&#91;</span>36<span class="cite-bracket">&#93;</span></a></sup> The <code>docker swarm</code> CLI<sup id="cite_ref-Auto38-3_37-0" class="reference"><a href="#cite_note-Auto38-3-37"><span class="cite-bracket">&#91;</span>37<span class="cite-bracket">&#93;</span></a></sup> utility allows users to run Swarm containers, create discovery tokens, list nodes in the cluster, and more.<sup id="cite_ref-swarm-cli-ref_38-0" class="reference"><a href="#cite_note-swarm-cli-ref-38"><span class="cite-bracket">&#91;</span>38<span class="cite-bracket">&#93;</span></a></sup> The <code>docker node</code> CLI utility allows users to run various commands to manage nodes in a swarm, for example, listing the nodes in a swarm, updating nodes, and removing nodes from the swarm.<sup id="cite_ref-docker-node_39-0" class="reference"><a href="#cite_note-docker-node-39"><span class="cite-bracket">&#91;</span>39<span class="cite-bracket">&#93;</span></a></sup> Docker manages swarms using the <a href="/wiki/Raft_(computer_science)" class="mw-redirect" title="Raft (computer science)">Raft</a> <a href="/wiki/Consensus_(computer_science)" title="Consensus (computer science)">consensus</a> algorithm. According to Raft, for an update to be performed, the majority of Swarm nodes need to agree on the update.<sup id="cite_ref-aquasec_40-0" class="reference"><a href="#cite_note-aquasec-40"><span class="cite-bracket">&#91;</span>40<span class="cite-bracket">&#93;</span></a></sup><sup id="cite_ref-raft_41-0" class="reference"><a href="#cite_note-raft-41"><span class="cite-bracket">&#91;</span>41<span class="cite-bracket">&#93;</span></a></sup> In addition to the <code>docker swarm</code> CLI, <code>docker stack</code> is a tool designed to manage Swarm services with greater flexibility. It can use a configuration file very similar to a <b>docker-compose.yml</b>, with a few nuances. Using <code>docker stack</code> instead of <code>docker compose</code> offers several advantages, such as the ability to manage a Swarm cluster across multiple machines or the capability to work with <code>docker secret</code> combined with <code>docker context</code>, a feature that allows executing Docker commands on a remote host, enabling remote container management.</li></ul>
<ul><li><b>Docker Volume</b> facilitates the independent persistence of data, allowing data to remain even after the container is deleted or re-created.<sup id="cite_ref-myunidockerguide2_42-0" class="reference"><a href="#cite_note-myunidockerguide2-42"><span class="cite-bracket">&#91;</span>42<span class="cite-bracket">&#93;</span></a></sup></li></ul>
<div class="mw-heading mw-heading2"><h2 id="History">History</h2><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Docker_(software)&amp;action=edit&amp;section=7" title="Edit section: History"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></div>
<p>dotCloud Inc. was founded by Kamel Founadi, Solomon Hykes, and Sebastien Pahl<sup id="cite_ref-Auto38-4_43-0" class="reference"><a href="#cite_note-Auto38-4-43"><span class="cite-bracket">&#91;</span>43<span class="cite-bracket">&#93;</span></a></sup> during the <a href="/wiki/Y_Combinator" title="Y Combinator">Y Combinator</a> Summer 2010 startup incubator group and launched in 2011, and renamed to Docker Inc in 2013.<sup id="cite_ref-Auto38-5_44-0" class="reference"><a href="#cite_note-Auto38-5-44"><span class="cite-bracket">&#91;</span>44<span class="cite-bracket">&#93;</span></a></sup> The startup was also one of the 12 startups in <i>Founder's Den</i> first cohort.<sup id="cite_ref-Auto38-6_45-0" class="reference"><a href="#cite_note-Auto38-6-45"><span class="cite-bracket">&#91;</span>45<span class="cite-bracket">&#93;</span></a></sup> Hykes started the Docker project in France as an internal project within <a href="/wiki/DotCloud" class="mw-redirect" title="DotCloud">dotCloud</a>, a <a href="/wiki/Platform_as_a_service" title="Platform as a service">platform-as-a-service</a> company.<sup id="cite_ref-dotcloud_46-0" class="reference"><a href="#cite_note-dotcloud-46"><span class="cite-bracket">&#91;</span>46<span class="cite-bracket">&#93;</span></a></sup>
</p><p>Docker debuted to the public in <a href="/wiki/Santa_Clara,_California" title="Santa Clara, California">Santa Clara</a> at <a href="/wiki/PyCon" class="mw-redirect" title="PyCon">PyCon</a> in 2013.<sup id="cite_ref-docker-announcement_47-0" class="reference"><a href="#cite_note-docker-announcement-47"><span class="cite-bracket">&#91;</span>47<span class="cite-bracket">&#93;</span></a></sup> It was released as <a href="/wiki/Open-source" class="mw-redirect" title="Open-source">open-source</a> in March 2013.<sup id="cite_ref-infoq-2013-03_20-1" class="reference"><a href="#cite_note-infoq-2013-03-20"><span class="cite-bracket">&#91;</span>20<span class="cite-bracket">&#93;</span></a></sup> At the time, it used <a href="/wiki/LXC" title="LXC">LXC</a> as its default execution environment. One year later, with the release of version 0.9, Docker replaced LXC with its own component, <i>libcontainer</i>, which was written in the <a href="/wiki/Go_(programming_language)" title="Go (programming language)">Go</a> programming language.<sup id="cite_ref-zdnet-7000030397_18-1" class="reference"><a href="#cite_note-zdnet-7000030397-18"><span class="cite-bracket">&#91;</span>18<span class="cite-bracket">&#93;</span></a></sup><sup id="cite_ref-infoq-2014-03_48-0" class="reference"><a href="#cite_note-infoq-2014-03-48"><span class="cite-bracket">&#91;</span>48<span class="cite-bracket">&#93;</span></a></sup>
</p><p>In 2017, Docker created the <a href="/w/index.php?title=Moby_project&amp;action=edit&amp;redlink=1" class="new" title="Moby project (page does not exist)">Moby project</a><span class="noprint" style="font-size:85%; font-style: normal;">&#160;&#91;<a href="https://www.wikidata.org/wiki/Special:EntityPage/Q96276695#sitelinks-wikipedia" class="extiw" title="d:Special:EntityPage/Q96276695"><span title="Wikidata list: &quot;Moby project&quot; articles in other languages">d</span></a>&#93;</span> for open research and development.<sup id="cite_ref-moby_49-0" class="reference"><a href="#cite_note-moby-49"><span class="cite-bracket">&#91;</span>49<span class="cite-bracket">&#93;</span></a></sup>
</p>

<li>December 4, 2014: <a href="/wiki/IBM" title="IBM">IBM</a> announced a strategic partnership with Docker that enables Docker to integrate more closely with the IBM Cloud.<sup id="cite_ref-ibm-partnership_55-0" class="reference"><a href="#cite_note-ibm-partnership-55"><span class="cite-bracket">&#91;</span>55<span class="cite-bracket">&#93;</span></a></sup></li>
<li>June 22, 2015: Docker and several other companies announced that they were working on a new vendor and operating-system-independent standard for software containers.<sup id="cite_ref-common-container-standard_56-0" class="reference"><a href="#cite_note-common-container-standard-56"><span class="cite-bracket">&#91;</span>56<span class="cite-bracket">&#93;</span></a></sup><sup id="cite_ref-open-container-project_57-0" class="reference"><a href="#cite_note-open-container-project-57"><span class="cite-bracket">&#91;</span>57<span class="cite-bracket">&#93;</span></a></sup></li>
<li>December 2015: <a href="/wiki/Oracle_Cloud" title="Oracle Cloud">Oracle Cloud</a> added Docker container support after acquiring <a href="/wiki/StackEngine" title="StackEngine">StackEngine</a>, a Docker container startup.<sup id="cite_ref-58" class="reference"><a href="#cite_note-58"><span class="cite-bracket">&#91;</span>58<span class="cite-bracket">&#93;</span></a></sup></li>
<li>April 2016: Windocks, an <a href="/wiki/Independent_software_vendor" title="Independent software vendor">independent software vendor</a> released a port of Docker's open source project to Windows, supporting Windows Server 2012 R2 and Server 2016, with all editions of SQL Server 2008 onward.<sup id="cite_ref-InfoWorld-Windocks_59-0" class="reference"><a href="#cite_note-InfoWorld-Windocks-59"><span class="cite-bracket">&#91;</span>59<span class="cite-bracket">&#93;</span></a></sup></li>
<li>May 2016: analysis showed the following organizations as main contributors to Docker: The Docker team, <a href="/wiki/Cisco" title="Cisco">Cisco</a>, <a href="/wiki/Google" title="Google">Google</a>, <a href="/wiki/Huawei" title="Huawei">Huawei</a>, <a href="/wiki/IBM" title="IBM">IBM</a>, <a href="/wiki/Microsoft" title="Microsoft">Microsoft</a>, and <a href="/wiki/Red_Hat" title="Red Hat">Red Hat</a>.<sup id="cite_ref-gist-project-stats_60-0" class="reference"><a href="#cite_note-gist-project-stats-60"><span class="cite-bracket">&#91;</span>60<span class="cite-bracket">&#93;</span></a></sup></li>
<li>June 8, 2016: Microsoft announced that Docker could now be used natively on <a href="/wiki/Windows_10" title="Windows 10">Windows 10</a>.<sup id="cite_ref-win10-insider-preview_61-0" class="reference"><a href="#cite_note-win10-insider-preview-61"><span class="cite-bracket">&#91;</span>61<span class="cite-bracket">&#93;</span></a></sup></li>
<li>January 2017: An analysis of <a href="/wiki/LinkedIn" title="LinkedIn">LinkedIn</a> profile mentions showed Docker presence grew by 160% in 2016.<sup id="cite_ref-linkedin-pulse_62-0" class="reference"><a href="#cite_note-linkedin-pulse-62"><span class="cite-bracket">&#91;</span>62<span class="cite-bracket">&#93;</span></a></sup></li>
<li>May 6, 2019: Microsoft announced the second version of <a href="/wiki/Windows_Subsystem_for_Linux" title="Windows Subsystem for Linux">Windows Subsystem for Linux</a> (WSL). Docker, Inc. announced that it had started working on a version of Docker for Windows to run on WSL 2.<sup id="cite_ref-Vaughan-Nichols_63-0" class="reference"><a href="#cite_note-Vaughan-Nichols-63"><span class="cite-bracket">&#91;</span>63<span class="cite-bracket">&#93;</span></a></sup> In particular, this meant Docker could run on Windows 10 Home (previously it was limited to Windows Pro and Enterprise since it used <a href="/wiki/Hyper-V" title="Hyper-V">Hyper-V</a>).</li>
<li>August 2020: Microsoft announced a backport of WSL2 to Windows 10 versions 1903 and 1909 (previously WSL2 was available only on version 2004)<sup id="cite_ref-WSL2-backport_64-0" class="reference"><a href="#cite_note-WSL2-backport-64"><span class="cite-bracket">&#91;</span>64<span class="cite-bracket">&#93;</span></a></sup> and Docker developers announced availability of Docker for these platforms.<sup id="cite_ref-docker-backport_65-0" class="reference"><a href="#cite_note-docker-backport-65"><span class="cite-bracket">&#91;</span>65<span class="cite-bracket">&#93;</span></a></sup></li>
<li>August 2021: Docker Desktop for Windows and MacOS was no longer available free of charge for enterprise users. Docker ended free Docker Desktop use for larger business customers and replaced its Free Plan with a Personal Plan. Docker on Linux distributions remained unaffected.<sup id="cite_ref-66" class="reference"><a href="#cite_note-66"><span class="cite-bracket">&#91;</span>66<span class="cite-bracket">&#93;</span></a></sup></li>
<li>December 2023: Docker acquired AtomicJar to expand its testing capabilities.<sup id="cite_ref-67" class="reference"><a href="#cite_note-67"><span class="cite-bracket">&#91;</span>67<span class="cite-bracket">&#93;</span></a></sup></li></ul>
