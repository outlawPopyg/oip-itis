<p><b>MongoDB</b> is a <a href="/wiki/Source-available" class="mw-redirect" title="Source-available">source-available</a>, <a href="/wiki/Cross-platform" class="mw-redirect" title="Cross-platform">cross-platform</a>, <a href="/wiki/Document-oriented_database" title="Document-oriented database">document-oriented database</a> program. Classified as a <a href="/wiki/NoSQL" title="NoSQL">NoSQL</a> database product, MongoDB uses <a href="/wiki/JSON" title="JSON">JSON</a>-like documents with optional <a href="/wiki/Database_schema" title="Database schema">schemas</a>. Released in February 2009 by <a href="/wiki/10gen" class="mw-redirect" title="10gen">10gen</a> (now <a href="/wiki/MongoDB_Inc." title="MongoDB Inc.">MongoDB Inc.</a>), it supports features like <a href="/wiki/Sharding" class="mw-redirect" title="Sharding">sharding</a>, <a href="/wiki/Replication_(computing)" title="Replication (computing)">replication</a>, and <a href="/wiki/ACID" title="ACID">ACID</a> transactions (from version 4.0). MongoDB Atlas, its managed cloud service, operates on <a href="/wiki/AWS" class="mw-redirect" title="AWS">AWS</a>, <a href="/wiki/Google_Cloud_Platform" title="Google Cloud Platform">Google Cloud Platform</a>, and <a href="/wiki/Microsoft_Azure" title="Microsoft Azure">Microsoft Azure</a>. Current versions are licensed under the <a href="/wiki/Server_Side_Public_License" title="Server Side Public License">Server Side Public License</a> (SSPL). MongoDB is a member of the <a href="/wiki/MACH_Alliance" title="MACH Alliance">MACH Alliance</a>.
</p>

<div class="mw-heading mw-heading2"><h2 id="History">History</h2><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=MongoDB&amp;action=edit&amp;section=1" title="Edit section: History"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></div>
<style data-mw-deduplicate="TemplateStyles:r1236090951">.mw-parser-output .hatnote{font-style:italic}.mw-parser-output div.hatnote{padding-left:1.6em;margin-bottom:0.5em}.mw-parser-output .hatnote i{font-style:normal}.mw-parser-output .hatnote+link+.hatnote{margin-top:-0.5em}@media print{body.ns-0 .mw-parser-output .hatnote{display:none!important}}</style><div role="note" class="hatnote navigation-not-searchable">See also: <a href="/wiki/MongoDB_Inc.#History" title="MongoDB Inc.">MongoDB Inc. §&#160;History</a></div>
<p>The American software company 10gen began developing MongoDB in 2007 as a component of a planned <a href="/wiki/Platform_as_a_service" title="Platform as a service">platform-as-a-service</a> product. In 2009, the company shifted to an open-source development model and began offering commercial support and other services. In 2013, 10gen changed its name to MongoDB Inc.<sup id="cite_ref-gigaom-rename_5-0" class="reference"><a href="#cite_note-gigaom-rename-5"><span class="cite-bracket">&#91;</span>5<span class="cite-bracket">&#93;</span></a></sup>
</p><p>On October 20, 2017, MongoDB became a publicly traded company, listed on NASDAQ as MDB with an IPO price of $24 per share.<sup id="cite_ref-6" class="reference"><a href="#cite_note-6"><span class="cite-bracket">&#91;</span>6<span class="cite-bracket">&#93;</span></a></sup>
</p><p>On November 8, 2018, with the stable release 4.0.4,  the software's license changed from AGPL 3.0 to SSPL.<sup id="cite_ref-7" class="reference"><a href="#cite_note-7"><span class="cite-bracket">&#91;</span>7<span class="cite-bracket">&#93;</span></a></sup><sup id="cite_ref-8" class="reference"><a href="#cite_note-8"><span class="cite-bracket">&#91;</span>8<span class="cite-bracket">&#93;</span></a></sup>
</p><p>On October 30, 2019, MongoDB teamed with <a href="/wiki/Alibaba_Cloud" title="Alibaba Cloud">Alibaba Cloud</a> to offer Alibaba Cloud customers a MongoDB-as-a-service solution. Customers can use the managed offering from Alibaba's global data centers.<sup id="cite_ref-9" class="reference"><a href="#cite_note-9"><span class="cite-bracket">&#91;</span>9<span class="cite-bracket">&#93;</span></a></sup>
</p>

<div class="mw-heading mw-heading2"><h2 id="Main_features">Main features</h2><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=MongoDB&amp;action=edit&amp;section=2" title="Edit section: Main features"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></div>
<div class="mw-heading mw-heading3"><h3 id="Ad-hoc_queries">Ad-hoc queries</h3><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=MongoDB&amp;action=edit&amp;section=3" title="Edit section: Ad-hoc queries"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></div>
<p>MongoDB supports field, <a href="/wiki/Range_query_(database)" title="Range query (database)">range query</a> and <a href="/wiki/Regular_expression" title="Regular expression">regular-expression</a> searches.<sup id="cite_ref-MongoDB_find_32-0" class="reference"><a href="#cite_note-MongoDB_find-32"><span class="cite-bracket">&#91;</span>32<span class="cite-bracket">&#93;</span></a></sup> Queries can return specific fields of documents and also include user-defined <a href="/wiki/JavaScript" title="JavaScript">JavaScript</a> functions. Queries can also be configured to return a random sample of results of a given size.
</p>
<div class="mw-heading mw-heading3"><h3 id="Indexing">Indexing</h3><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=MongoDB&amp;action=edit&amp;section=4" title="Edit section: Indexing"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></div>
<p>Fields in a MongoDB document can be indexed with primary and secondary indices.
</p>
<div class="mw-heading mw-heading3"><h3 id="Replication">Replication</h3><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=MongoDB&amp;action=edit&amp;section=5" title="Edit section: Replication"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></div>
<p>MongoDB provides high availability with replica sets.<sup id="cite_ref-33" class="reference"><a href="#cite_note-33"><span class="cite-bracket">&#91;</span>33<span class="cite-bracket">&#93;</span></a></sup> A replica set consists of two or more copies of the data. Each replica-set member may act in the role of primary or secondary replica at any time. All writes and reads are done on the primary replica by default. Secondary replicas maintain a copy of the data of the primary using built-in replication. When a primary replica fails, the replica set automatically conducts an election process to determine which secondary should become the primary. Secondaries can optionally serve read operations, but that data is only eventually consistent by default.
</p><p>If the replicated MongoDB deployment only has a single secondary member, a separate <a href="/wiki/Daemon_(computing)" title="Daemon (computing)">daemon</a> called an arbiter must be added to the set. It has the single responsibility of resolving the election of the new primary.<sup id="cite_ref-ARBY_34-0" class="reference"><a href="#cite_note-ARBY-34"><span class="cite-bracket">&#91;</span>34<span class="cite-bracket">&#93;</span></a></sup> As a consequence, an ideal distributed MongoDB deployment requires at least three separate servers, even in the case of just one primary and one secondary.<sup id="cite_ref-ARBY_34-1" class="reference"><a href="#cite_note-ARBY-34"><span class="cite-bracket">&#91;</span>34<span class="cite-bracket">&#93;</span></a></sup>
</p>
<div class="mw-heading mw-heading3"><h3 id="Load_balancing">Load balancing</h3><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=MongoDB&amp;action=edit&amp;section=6" title="Edit section: Load balancing"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></div>
<p>MongoDB scales horizontally using <a href="/wiki/Sharding" class="mw-redirect" title="Sharding">sharding</a>.<sup id="cite_ref-35" class="reference"><a href="#cite_note-35"><span class="cite-bracket">&#91;</span>35<span class="cite-bracket">&#93;</span></a></sup> The user chooses a shard key, which determines how the data in a collection will be distributed. The data is split into ranges (based on the shard key) and distributed across multiple shards, which are masters with one or more replicas. Alternatively, the shard key can be hashed to map to a shard – enabling an even data distribution.
</p><p>MongoDB can run over multiple servers, <a href="/wiki/Load_balancing_(computing)" title="Load balancing (computing)">balancing the load</a> or duplicating data to keep the system functional in case of hardware failure.
</p>
<div class="mw-heading mw-heading3"><h3 id="File_storage">File storage</h3><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=MongoDB&amp;action=edit&amp;section=7" title="Edit section: File storage"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></div>
<p>MongoDB can be used as a <a href="/wiki/File_system" title="File system">file system</a>, called <a href="/w/index.php?title=GridFS&amp;action=edit&amp;redlink=1" class="new" title="GridFS (page does not exist)">GridFS</a>, with load-balancing and data-replication features over multiple machines for storing files.
</p><p>This function, called a <a href="/wiki/Grid_file_system" title="Grid file system">grid file system</a>,<sup id="cite_ref-36" class="reference"><a href="#cite_note-36"><span class="cite-bracket">&#91;</span>36<span class="cite-bracket">&#93;</span></a></sup> is included with MongoDB drivers. MongoDB exposes functions for file manipulation and content to developers. GridFS can be accessed using the mongofiles utility or plugins for <a href="/wiki/Nginx" title="Nginx">Nginx</a><sup id="cite_ref-37" class="reference"><a href="#cite_note-37"><span class="cite-bracket">&#91;</span>37<span class="cite-bracket">&#93;</span></a></sup> and <a href="/wiki/Lighttpd" title="Lighttpd">lighttpd</a>.<sup id="cite_ref-38" class="reference"><a href="#cite_note-38"><span class="cite-bracket">&#91;</span>38<span class="cite-bracket">&#93;</span></a></sup> GridFS divides a file into parts, or chunks, and stores each of those chunks as a separate document.<sup id="cite_ref-39" class="reference"><a href="#cite_note-39"><span class="cite-bracket">&#91;</span>39<span class="cite-bracket">&#93;</span></a></sup>
</p>
<div class="mw-heading mw-heading3"><h3 id="Aggregation">Aggregation</h3><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=MongoDB&amp;action=edit&amp;section=8" title="Edit section: Aggregation"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></div>
<p>MongoDB provides three ways to perform aggregation: the aggregation pipeline, the map-reduce function and single-purpose aggregation methods.<sup id="cite_ref-40" class="reference"><a href="#cite_note-40"><span class="cite-bracket">&#91;</span>40<span class="cite-bracket">&#93;</span></a></sup>
</p><p><a href="/wiki/Map-reduce" class="mw-redirect" title="Map-reduce">Map-reduce</a> can be used for batch processing of data and aggregation operations. However, according to MongoDB's documentation, the aggregation pipeline provides better performance for most aggregation operations.<sup id="cite_ref-41" class="reference"><a href="#cite_note-41"><span class="cite-bracket">&#91;</span>41<span class="cite-bracket">&#93;</span></a></sup>
</p><p>The aggregation framework enables users to obtain results similar to those returned by queries that include the <a href="/wiki/SQL" title="SQL">SQL</a> GROUP BY clause. Aggregation operators can be strung together to form a pipeline, analogous to <a href="/wiki/Pipeline_(Unix)" title="Pipeline (Unix)">Unix pipes</a>. The aggregation framework includes the $lookup operator, which can join documents from multiple collections, as well as statistical operators such as standard deviation.
</p>
<div class="mw-heading mw-heading3"><h3 id="Server-side_JavaScript_execution">Server-side JavaScript execution</h3><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=MongoDB&amp;action=edit&amp;section=9" title="Edit section: Server-side JavaScript execution"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></div>
<p>JavaScript can be used in queries, aggregation functions (such as <a href="/wiki/MapReduce" title="MapReduce">MapReduce</a>) and sent directly to the database to be executed.
</p>
<div class="mw-heading mw-heading3"><h3 id="Capped_collections">Capped collections</h3><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=MongoDB&amp;action=edit&amp;section=10" title="Edit section: Capped collections"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></div>
<p>MongoDB supports fixed-size collections called capped collections. This type of collection maintains insertion order and, once the specified size has been reached, behaves like a <a href="/wiki/Circular_queue" class="mw-redirect" title="Circular queue">circular queue</a>.
</p>
<div class="mw-heading mw-heading3"><h3 id="Transactions">Transactions</h3><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=MongoDB&amp;action=edit&amp;section=11" title="Edit section: Transactions"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></div>
<p>MongoDB supports multi-document <a href="/wiki/ACID" title="ACID">ACID</a> transactions since the 4.0 release in June 2018.<sup id="cite_ref-42" class="reference"><a href="#cite_note-42"><span class="cite-bracket">&#91;</span>42<span class="cite-bracket">&#93;</span></a></sup>
</p>
<div class="mw-heading mw-heading2"><h2 id="Editions">Editions</h2><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=MongoDB&amp;action=edit&amp;section=12" title="Edit section: Editions"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></div>

<p>The MongoDB Community Edition is free and available for Windows, Linux and macOS.<sup id="cite_ref-43" class="reference"><a href="#cite_note-43"><span class="cite-bracket">&#91;</span>43<span class="cite-bracket">&#93;</span></a></sup>
</p>
<div class="mw-heading mw-heading3"><h3 id="MongoDB_Enterprise_Server">MongoDB Enterprise Server</h3><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=MongoDB&amp;action=edit&amp;section=14" title="Edit section: MongoDB Enterprise Server"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></div>

</p>
<div class="mw-heading mw-heading3"><h3 id="MongoDB_Atlas">MongoDB Atlas</h3><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=MongoDB&amp;action=edit&amp;section=15" title="Edit section: MongoDB Atlas"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></div>
<p>MongoDB is also available as an on-demand, fully managed service. MongoDB Atlas runs on AWS, Microsoft Azure and Google Cloud Platform.<sup id="cite_ref-45" class="reference"><a href="#cite_note-45"><span class="cite-bracket">&#91;</span>45<span class="cite-bracket">&#93;</span></a></sup>
</p><p>On March 10, 2022, MongoDB warned its users in Russia and <a href="/wiki/Belarus" title="Belarus">Belarus</a> that their data stored on the MongoDB Atlas platform will be destroyed as a result of American sanctions related to the <a href="/wiki/Russo-Ukrainian_War" title="Russo-Ukrainian War">Russo-Ukrainian War</a>.<sup id="cite_ref-46" class="reference"><a href="#cite_note-46"><span class="cite-bracket">&#91;</span>46<span class="cite-bracket">&#93;</span></a></sup>
</p>
<div class="mw-heading mw-heading2"><h2 id="Architecture">Architecture</h2><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=MongoDB&amp;action=edit&amp;section=16" title="Edit section: Architecture"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></div>
<div class="mw-heading mw-heading3"><h3 id="Programming_language_accessibility">Programming language accessibility</h3><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=MongoDB&amp;action=edit&amp;section=17" title="Edit section: Programming language accessibility"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></div>
<p>MongoDB has official drivers for major programming languages and development environments.<sup id="cite_ref-github.io_47-0" class="reference"><a href="#cite_note-github.io-47"><span class="cite-bracket">&#91;</span>47<span class="cite-bracket">&#93;</span></a></sup> There are also a large number of unofficial or community-supported drivers for other programming languages and frameworks.
</p>
<div class="mw-heading mw-heading3"><h3 id="Serverless_access">Serverless access</h3><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=MongoDB&amp;action=edit&amp;section=18" title="Edit section: Serverless access"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></div>
<div class="mw-heading mw-heading3"><h3 id="Management_and_graphical_front-ends">Management and graphical front-ends</h3><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=MongoDB&amp;action=edit&amp;section=19" title="Edit section: Management and graphical front-ends"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></div>

<p>The primary interface to the database has been the mongo shell. Since MongoDB 3.2, MongoDB Compass is introduced as the native GUI. There are products and third-party projects that offer user interfaces for administration and data viewing.<sup id="cite_ref-48" class="reference"><a href="#cite_note-48"><span class="cite-bracket">&#91;</span>48<span class="cite-bracket">&#93;</span></a></sup>
</p>
<div class="mw-heading mw-heading2"><h2 id="Licensing">Licensing</h2><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=MongoDB&amp;action=edit&amp;section=20" title="Edit section: Licensing"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></div>
<div class="mw-heading mw-heading3"><h3 id="MongoDB_Community_Server_2">MongoDB Community Server</h3><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=MongoDB&amp;action=edit&amp;section=21" title="Edit section: MongoDB Community Server"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></div>
<p>As of October 2018, MongoDB is released under the <a href="/wiki/Server_Side_Public_License" title="Server Side Public License">Server Side Public License</a> (SSPL), a non-free license developed by the project. It replaces the <a href="/wiki/GNU_Affero_General_Public_License" title="GNU Affero General Public License">GNU Affero General Public License</a>, and is nearly identical to the <a href="/wiki/GNU_General_Public_License_version_3" class="mw-redirect" title="GNU General Public License version 3">GNU General Public License version 3</a>, but requires that those making the software publicly available as part of a "service" must make the service's entire source code (insofar that a user would be able to recreate the service themselves) available under this license. By contrast, the AGPL only requires the source code of the licensed software to be provided to users when the software is conveyed over a network.<sup id="cite_ref-49" class="reference"><a href="#cite_note-49"><span class="cite-bracket">&#91;</span>49<span class="cite-bracket">&#93;</span></a></sup><sup id="cite_ref-50" class="reference"><a href="#cite_note-50"><span class="cite-bracket">&#91;</span>50<span class="cite-bracket">&#93;</span></a></sup> The SSPL was submitted for certification to the <a href="/wiki/Open_Source_Initiative" title="Open Source Initiative">Open Source Initiative</a> but later withdrawn.<sup id="cite_ref-51" class="reference"><a href="#cite_note-51"><span class="cite-bracket">&#91;</span>51<span class="cite-bracket">&#93;</span></a></sup> In January 2021, the Open Source Initiative stated that SSPL is not an open source license.<sup id="cite_ref-52" class="reference"><a href="#cite_note-52"><span class="cite-bracket">&#91;</span>52<span class="cite-bracket">&#93;</span></a></sup> The language drivers are available under an <a href="/wiki/Apache_License" title="Apache License">Apache License</a>. In addition, MongoDB Inc. offers proprietary licenses for MongoDB. The last versions licensed as AGPL version 3 are 4.0.3 (stable) and 4.1.4.<sup id="cite_ref-53" class="reference"><a href="#cite_note-53"><span class="cite-bracket">&#91;</span>53<span class="cite-bracket">&#93;</span></a></sup>
</p><p>MongoDB has been removed from the <a href="/wiki/Debian" title="Debian">Debian</a>, <a href="/wiki/Fedora_(operating_system)" class="mw-redirect" title="Fedora (operating system)">Fedora</a> and <a href="/wiki/Red_Hat_Enterprise_Linux" title="Red Hat Enterprise Linux">Red Hat Enterprise Linux</a> distributions because of the licensing change. Fedora determined that the SSPL version 1 is not a free software license because it is "intentionally crafted to be aggressively discriminatory" towards commercial users.<sup id="cite_ref-54" class="reference"><a href="#cite_note-54"><span class="cite-bracket">&#91;</span>54<span class="cite-bracket">&#93;</span></a></sup><sup id="cite_ref-55" class="reference"><a href="#cite_note-55"><span class="cite-bracket">&#91;</span>55<span class="cite-bracket">&#93;</span></a></sup>
</p>
<div class="mw-heading mw-heading2"><h2 id="Bug_reports_and_criticisms">Bug reports and criticisms</h2><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=MongoDB&amp;action=edit&amp;section=22" title="Edit section: Bug reports and criticisms"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></div>
<div class="mw-heading mw-heading3"><h3 id="Security">Security</h3><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=MongoDB&amp;action=edit&amp;section=23" title="Edit section: Security"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></div>
<p>Because of MongoDB's default security configuration, which allows any user full access to the database, data from tens of thousands of MongoDB installations has been stolen. Furthermore, many MongoDB servers have been <a href="/wiki/Ransomware" title="Ransomware">held for ransom</a>.<sup id="cite_ref-56" class="reference"><a href="#cite_note-56"><span class="cite-bracket">&#91;</span>56<span class="cite-bracket">&#93;</span></a></sup><sup id="cite_ref-57" class="reference"><a href="#cite_note-57"><span class="cite-bracket">&#91;</span>57<span class="cite-bracket">&#93;</span></a></sup> In September 2017, Davi Ottenheimer head of product security at MongoDB, proclaimed that measures had been taken to defend against these risks.<sup id="cite_ref-58" class="reference"><a href="#cite_note-58"><span class="cite-bracket">&#91;</span>58<span class="cite-bracket">&#93;</span></a></sup>
</p><p>From the MongoDB 2.6 release onward, the binaries for the official MongoDB RPM and DEB packages bind to <a href="/wiki/Localhost" title="Localhost">localhost</a> by default. From MongoDB 3.6, this default behavior was extended to all MongoDB packages across all platforms. As a result, all networked connections to the database are denied unless explicitly configured by an administrator.<sup id="cite_ref-59" class="reference"><a href="#cite_note-59"><span class="cite-bracket">&#91;</span>59<span class="cite-bracket">&#93;</span></a></sup>
</p>
<div class="mw-heading mw-heading3"><h3 id="Technical_criticisms">Technical criticisms</h3><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=MongoDB&amp;action=edit&amp;section=24" title="Edit section: Technical criticisms"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></div>
<p>In some failure scenarios in which an application can access two distinct MongoDB processes that cannot access each other, it is possible for MongoDB to return stale reads. It is also possible for MongoDB to roll back writes that have been acknowledged.<sup id="cite_ref-60" class="reference"><a href="#cite_note-60"><span class="cite-bracket">&#91;</span>60<span class="cite-bracket">&#93;</span></a></sup> The issue was addressed in version 3.4.0, released in November 2016,<sup id="cite_ref-61" class="reference"><a href="#cite_note-61"><span class="cite-bracket">&#91;</span>61<span class="cite-bracket">&#93;</span></a></sup> and applied to earlier releases from v3.2.12 onward.<sup id="cite_ref-62" class="reference"><a href="#cite_note-62"><span class="cite-bracket">&#91;</span>62<span class="cite-bracket">&#93;</span></a></sup>
</p><p>Before version 2.2, <a href="/wiki/Lock_(database)" class="mw-redirect" title="Lock (database)">locks</a> were implemented on a per-server-process basis. With version 2.2, locks were implemented at the database level.<sup id="cite_ref-63" class="reference"><a href="#cite_note-63"><span class="cite-bracket">&#91;</span>63<span class="cite-bracket">&#93;</span></a></sup> Beginning with version 3.0,<sup id="cite_ref-auto_64-0" class="reference"><a href="#cite_note-auto-64"><span class="cite-bracket">&#91;</span>64<span class="cite-bracket">&#93;</span></a></sup> pluggable storage engines are available, and each storage engine may implement locks differently.<sup id="cite_ref-auto_64-1" class="reference"><a href="#cite_note-auto-64"><span class="cite-bracket">&#91;</span>64<span class="cite-bracket">&#93;</span></a></sup> With MongoDB 3.0, locks are implemented at the collection level for the MMAPv1 storage engine,<sup id="cite_ref-65" class="reference"><a href="#cite_note-65"><span class="cite-bracket">&#91;</span>65<span class="cite-bracket">&#93;</span></a></sup> while the <a href="/wiki/WiredTiger" title="WiredTiger">WiredTiger</a> storage engine uses an optimistic concurrency protocol that effectively provides document-level locking.<sup id="cite_ref-66" class="reference"><a href="#cite_note-66"><span class="cite-bracket">&#91;</span>66<span class="cite-bracket">&#93;</span></a></sup> Even with versions prior to 3.0, one approach to increase concurrency is to use <a href="/wiki/Shard_(database_architecture)" title="Shard (database architecture)">sharding</a>.<sup id="cite_ref-67" class="reference"><a href="#cite_note-67"><span class="cite-bracket">&#91;</span>67<span class="cite-bracket">&#93;</span></a></sup> In some situations, reads and writes will yield their locks. If MongoDB predicts that a page is unlikely to be in memory, operations will yield their lock while the pages load. The use of lock yielding expanded greatly in version 2.2.<sup id="cite_ref-68" class="reference"><a href="#cite_note-68"><span class="cite-bracket">&#91;</span>68<span class="cite-bracket">&#93;</span></a></sup>
</p><p>Until version 3.3.11, MongoDB could not perform <a href="/wiki/Collation" title="Collation">collation</a>-based sorting and was limited to bytewise comparison via <a href="/wiki/Memcmp" class="mw-redirect" title="Memcmp">memcmp</a>, which would not provide correct ordering for many non-English languages when used with a <a href="/wiki/Unicode" title="Unicode">Unicode</a> encoding. The issue was fixed on August 23, 2016.
</p><p>Prior to MongoDB 4.0, queries against an index were not atomic. Documents that were updated while queries was running could be missed.<sup id="cite_ref-69" class="reference"><a href="#cite_note-69"><span class="cite-bracket">&#91;</span>69<span class="cite-bracket">&#93;</span></a></sup> The introduction of the snapshot read concern in MongoDB 4.0 eliminated this risk.<sup id="cite_ref-70" class="reference"><a href="#cite_note-70"><span class="cite-bracket">&#91;</span>70<span class="cite-bracket">&#93;</span></a></sup>
</p><p>MongoDB claimed that version 3.6.4 had passed "the industry's toughest data safety, correctness, and consistency tests" by Jepsen, and that "MongoDB offers among the strongest data consistency, correctness, and safety guarantees of any database available today."<sup id="cite_ref-MongoDB_May_8,_2020_71-0" class="reference"><a href="#cite_note-MongoDB_May_8,_2020-71"><span class="cite-bracket">&#91;</span>71<span class="cite-bracket">&#93;</span></a></sup> Jepsen, which describes itself as a "distributed systems safety research company," disputed both claims on Twitter, saying, "In that report, MongoDB lost data and violated causal by default." In its May 2020 report on MongoDB version 4.2.6, Jepsen wrote that MongoDB had only mentioned tests that version 3.6.4 had passed, and that version had 4.2.6 introduced more problems.<sup id="cite_ref-Allen_2020_k853_72-0" class="reference"><a href="#cite_note-Allen_2020_k853-72"><span class="cite-bracket">&#91;</span>72<span class="cite-bracket">&#93;</span></a></sup> Jepsen's test summary reads in part:
</p>
<blockquote><p>Jepsen evaluated MongoDB version 4.2.6, and found that even at the strongest levels of read and write concern, it failed to preserve snapshot isolation. Instead, Jepsen observed read skew, cyclic information flow, duplicate writes, and internal consistency violations. Weak defaults meant that transactions could lose writes and allow dirty reads, even downgrading requested safety levels at the database and collection level. Moreover, the snapshot read concern did not guarantee snapshot unless paired with write concern majority—even for read-only transactions. These design choices complicate the safe use of MongoDB transactions.<sup id="cite_ref-Kingsbury_2020_y190_73-0" class="reference"><a href="#cite_note-Kingsbury_2020_y190-73"><span class="cite-bracket">&#91;</span>73<span class="cite-bracket">&#93;</span></a></sup></p></blockquote>
<p>On May 26, Jepsen updated the report to say: "MongoDB identified a bug in the transaction retry mechanism which they believe was responsible for the anomalies observed in this report; a patch is scheduled for 4.2.8."<sup id="cite_ref-Kingsbury_2020_y190_73-1" class="reference"><a href="#cite_note-Kingsbury_2020_y190-73"><span class="cite-bracket">&#91;</span>73<span class="cite-bracket">&#93;</span></a></sup> The issue has been patched as of that version, and "Jepsen criticisms of the default write concerns have also been addressed, with the default write concern now elevated to the majority concern (w:majority) from MongoDB 5.0."<sup id="cite_ref-MongoDB_June_10,_2023_74-0" class="reference"><a href="#cite_note-MongoDB_June_10,_2023-74"><span class="cite-bracket">&#91;</span>74<span class="cite-bracket">&#93;</span></a></sup>
</p>
