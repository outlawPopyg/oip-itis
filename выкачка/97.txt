					<div id="mw-content-text" class="mw-body-content"><div class="mw-content-ltr mw-parser-output" lang="en" dir="ltr"><div class="shortdescription nomobile noexcerpt noprint searchaux" style="display:none">Design pattern in object-oriented software development</div>
<figure class="mw-default-size" typeof="mw:File/Thumb"><a href="/wiki/File:Singleton_UML_class_diagram.svg" class="mw-file-description"><img src="//upload.wikimedia.org/wikipedia/commons/thumb/f/fb/Singleton_UML_class_diagram.svg/220px-Singleton_UML_class_diagram.svg.png" decoding="async" width="220" height="132" class="mw-file-element" srcset="//upload.wikimedia.org/wikipedia/commons/thumb/f/fb/Singleton_UML_class_diagram.svg/330px-Singleton_UML_class_diagram.svg.png 1.5x, //upload.wikimedia.org/wikipedia/commons/thumb/f/fb/Singleton_UML_class_diagram.svg/440px-Singleton_UML_class_diagram.svg.png 2x" data-file-width="500" data-file-height="300" /></a><figcaption>A <a href="/wiki/Class_diagram" title="Class diagram">class diagram</a> exemplifying the singleton pattern.</figcaption></figure>
<p>In <a href="/wiki/Object-oriented_programming" title="Object-oriented programming">object-oriented programming</a>, the <b>singleton pattern</b> is a <a href="/wiki/Software_design_pattern" title="Software design pattern">software design pattern</a> that restricts the <a href="/wiki/Instantiation_(computer_science)" class="mw-redirect" title="Instantiation (computer science)">instantiation</a> of a <a href="/wiki/Class_(computer_programming)" title="Class (computer programming)">class</a> to a singular instance. It is one of the well-known <a href="/wiki/Design_Patterns" title="Design Patterns">"Gang of Four" design patterns</a>, which describe how to solve recurring problems in object-oriented software.<sup id="cite_ref-GoF_1-0" class="reference"><a href="#cite_note-GoF-1"><span class="cite-bracket">&#91;</span>1<span class="cite-bracket">&#93;</span></a></sup> The pattern is useful when exactly one object is needed to coordinate actions across a system.
</p><p>More specifically, the singleton pattern allows classes to:<sup id="cite_ref-2" class="reference"><a href="#cite_note-2"><span class="cite-bracket">&#91;</span>2<span class="cite-bracket">&#93;</span></a></sup>
</p>
<ul><li>Ensure they only have one instance</li>
<li>Provide easy access to that instance</li>
<li>Control their instantiation (for example, hiding the <a href="/wiki/Constructor_(object-oriented_programming)" title="Constructor (object-oriented programming)">constructors</a> of a <a href="/wiki/Class_(computer_programming)" title="Class (computer programming)">class</a>)</li></ul>
<p>The term comes from the <a href="/wiki/Singleton_(mathematics)" title="Singleton (mathematics)">mathematical concept of a singleton</a>.
</p>
<meta property="mw:PageProp/toc" />
<div class="mw-heading mw-heading2"><h2 id="Common_uses">Common uses</h2><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Singleton_pattern&amp;action=edit&amp;section=1" title="Edit section: Common uses"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></div>
<p>Singletons are often preferred to <a href="/wiki/Global_variables" class="mw-redirect" title="Global variables">global variables</a> because they do not pollute the global <a href="/wiki/Namespace" title="Namespace">namespace</a> (or their containing namespace). Additionally, they permit <a href="/wiki/Lazy_evaluation" title="Lazy evaluation">lazy</a> allocation and initialization, whereas global variables in many languages will always consume resources.<sup id="cite_ref-GoF_1-1" class="reference"><a href="#cite_note-GoF-1"><span class="cite-bracket">&#91;</span>1<span class="cite-bracket">&#93;</span></a></sup><sup id="cite_ref-devin_3-0" class="reference"><a href="#cite_note-devin-3"><span class="cite-bracket">&#91;</span>3<span class="cite-bracket">&#93;</span></a></sup>
</p><p>The singleton pattern can also be used as a basis for other design patterns, such as the <a href="/wiki/Abstract_factory_pattern" title="Abstract factory pattern">abstract factory</a>, <a href="/wiki/Factory_method_pattern" title="Factory method pattern">factory method</a>, <a href="/wiki/Builder_pattern" title="Builder pattern">builder</a> and <a href="/wiki/Prototype_pattern" title="Prototype pattern">prototype</a> patterns. <a href="/wiki/Facade_pattern" title="Facade pattern">Facade</a> objects are also often singletons because only one facade object is required.
</p><p><a href="/wiki/Log_file" class="mw-redirect" title="Log file">Logging</a> is a common real-world use case for singletons, because all objects that wish to log messages require a uniform point of access and conceptually write to a single source.<sup id="cite_ref-rainsberger_4-0" class="reference"><a href="#cite_note-rainsberger-4"><span class="cite-bracket">&#91;</span>4<span class="cite-bracket">&#93;</span></a></sup>
</p>
<div class="mw-heading mw-heading2"><h2 id="Implementations">Implementations</h2><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Singleton_pattern&amp;action=edit&amp;section=2" title="Edit section: Implementations"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></div>
<p>Implementations of the singleton pattern ensure that only one instance of the singleton class ever exists and typically provide <a href="/wiki/Global_scope" class="mw-redirect" title="Global scope">global access</a> to that instance.
</p><p>Typically, this is accomplished by:
</p>
<ul><li>Declaring all <a href="/wiki/Constructor_(object-oriented_programming)" title="Constructor (object-oriented programming)">constructors</a> of the class to be <a href="/wiki/Private_member" class="mw-redirect" title="Private member">private</a>, which prevents it from being instantiated by other objects</li>
<li>Providing a <a href="/wiki/Static_method" class="mw-redirect" title="Static method">static method</a> that returns a <a href="/wiki/Reference_(computer_science)" title="Reference (computer science)">reference</a> to the instance</li></ul>
<p>The instance is usually stored as a private <a href="/wiki/Static_variable" title="Static variable">static variable</a>; the instance is created when the variable is initialized, at some point before when the static method is first called.
</p><p>This <a href="/wiki/C%2B%2B23" title="C++23">C++23</a> implementation is based on the pre-C++98 implementation in the book <sup class="noprint Inline-Template Template-Fact" style="white-space:nowrap;">&#91;<i><a href="/wiki/Wikipedia:Citation_needed" title="Wikipedia:Citation needed"><span title="the book is ambiguous here. (March 2024)">citation needed</span></a></i>&#93;</sup>.
</p>

<div class="mw-heading mw-heading3"><h3 id="Lazy_initialization">Lazy initialization</h3><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Singleton_pattern&amp;action=edit&amp;section=3" title="Edit section: Lazy initialization"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></div>
<p>A singleton implementation may use <a href="/wiki/Lazy_initialization" title="Lazy initialization">lazy initialization</a> in which the instance is created when the static method is first invoked. In <a href="/wiki/Multithreading_(software)" class="mw-redirect" title="Multithreading (software)">multithreaded</a> programs, this can cause <a href="/wiki/Race_condition" title="Race condition">race conditions</a> that result in the creation of multiple instances. The following <a href="/wiki/Java_version_history#Java_5" title="Java version history">Java 5+</a> example<sup id="cite_ref-6" class="reference"><a href="#cite_note-6"><span class="cite-bracket">&#91;</span>6<span class="cite-bracket">&#93;</span></a></sup> is a <a href="/wiki/Thread_safety" title="Thread safety">thread-safe</a> implementation, using lazy initialization with <a href="/wiki/Double-checked_locking" title="Double-checked locking">double-checked locking</a>.
</p>
<div class="mw-highlight mw-highlight-lang-java mw-content-ltr" dir="ltr"><pre><span></span><span class="kd">public</span><span class="w"> </span><span class="kd">class</span> <span class="nc">Singleton</span><span class="w"> </span><span class="p">{</span>

<span class="w">    </span><span class="kd">private</span><span class="w"> </span><span class="kd">static</span><span class="w"> </span><span class="kd">volatile</span><span class="w"> </span><span class="n">Singleton</span><span class="w"> </span><span class="n">instance</span><span class="w"> </span><span class="o">=</span><span class="w"> </span><span class="kc">null</span><span class="p">;</span>

<span class="w">    </span><span class="kd">private</span><span class="w"> </span><span class="nf">Singleton</span><span class="p">()</span><span class="w"> </span><span class="p">{}</span>

<span class="w">    </span><span class="kd">public</span><span class="w"> </span><span class="kd">static</span><span class="w"> </span><span class="n">Singleton</span><span class="w"> </span><span class="nf">getInstance</span><span class="p">()</span><span class="w"> </span><span class="p">{</span>
<span class="w">        </span><span class="k">if</span><span class="w"> </span><span class="p">(</span><span class="n">instance</span><span class="w"> </span><span class="o">==</span><span class="w"> </span><span class="kc">null</span><span class="p">)</span><span class="w"> </span><span class="p">{</span>
<span class="w">            </span><span class="kd">synchronized</span><span class="p">(</span><span class="n">Singleton</span><span class="p">.</span><span class="na">class</span><span class="p">)</span><span class="w"> </span><span class="p">{</span>
<span class="w">                </span><span class="k">if</span><span class="w"> </span><span class="p">(</span><span class="n">instance</span><span class="w"> </span><span class="o">==</span><span class="w"> </span><span class="kc">null</span><span class="p">)</span><span class="w"> </span><span class="p">{</span>
<span class="w">                    </span><span class="n">instance</span><span class="w"> </span><span class="o">=</span><span class="w"> </span><span class="k">new</span><span class="w"> </span><span class="n">Singleton</span><span class="p">();</span>
<span class="w">                </span><span class="p">}</span>
<span class="w">            </span><span class="p">}</span>
<span class="w">        </span><span class="p">}</span>
<span class="w">        </span><span class="k">return</span><span class="w"> </span><span class="n">instance</span><span class="p">;</span>
<span class="w">    </span><span class="p">}</span>
<span class="p">}</span>

<div class="mw-heading mw-heading2"><h2 id="Criticism">Criticism</h2><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Singleton_pattern&amp;action=edit&amp;section=4" title="Edit section: Criticism"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></div>
<p>Some consider the singleton to be an <a href="/wiki/Anti-pattern" title="Anti-pattern">anti-pattern</a> that introduces <a href="/wiki/Global_variables" class="mw-redirect" title="Global variables">global state</a> into an application, often unnecessarily. This introduces a potential dependency on the singleton by other objects, requiring analysis of implementation details to determine whether a dependency actually exists.<sup id="cite_ref-google_7-0" class="reference"><a href="#cite_note-google-7"><span class="cite-bracket">&#91;</span>7<span class="cite-bracket">&#93;</span></a></sup> This increased <a href="/wiki/Coupling_(computer_programming)" title="Coupling (computer programming)">coupling</a> can introduce difficulties with <a href="/wiki/Unit_testing" title="Unit testing">unit testing</a>.<sup id="cite_ref-button_8-0" class="reference"><a href="#cite_note-button-8"><span class="cite-bracket">&#91;</span>8<span class="cite-bracket">&#93;</span></a></sup> In turn, this places restrictions on any abstraction that uses the singleton, such as preventing <a href="/wiki/Concurrency_(computer_science)" title="Concurrency (computer science)">concurrent</a> use of multiple instances.<sup id="cite_ref-button_8-1" class="reference"><a href="#cite_note-button-8"><span class="cite-bracket">&#91;</span>8<span class="cite-bracket">&#93;</span></a></sup><sup id="cite_ref-9" class="reference"><a href="#cite_note-9"><span class="cite-bracket">&#91;</span>9<span class="cite-bracket">&#93;</span></a></sup><sup id="cite_ref-googletesting.blogspot.com_10-0" class="reference"><a href="#cite_note-googletesting.blogspot.com-10"><span class="cite-bracket">&#91;</span>10<span class="cite-bracket">&#93;</span></a></sup>
</p><p>Singletons also violate the <a href="/wiki/Single-responsibility_principle" title="Single-responsibility principle">single-responsibility principle</a> because they are responsible for enforcing their own uniqueness along with performing their normal functions.<sup id="cite_ref-button_8-2" class="reference"><a href="#cite_note-button-8"><span class="cite-bracket">&#91;</span>8<span class="cite-bracket">&#93;</span></a></sup>
</p>
