<p><b>Continuous delivery</b> (<b>CD</b>) is a <a href="/wiki/Software_engineering" title="Software engineering">software engineering</a> approach in which teams produce software in short cycles, ensuring that the software can be reliably released at any time.<sup id="cite_ref-CD_LC_1-0" class="reference"><a href="#cite_note-CD_LC-1"><span class="cite-bracket">&#91;</span>1<span class="cite-bracket">&#93;</span></a></sup><sup id="cite_ref-shahin-et-al-2017_2-0" class="reference"><a href="#cite_note-shahin-et-al-2017-2"><span class="cite-bracket">&#91;</span>2<span class="cite-bracket">&#93;</span></a></sup> It aims at building, testing, and releasing software with greater speed and frequency. The approach helps reduce the cost, time,<sup class="noprint Inline-Template Template-Fact" style="white-space:nowrap;">&#91;<i><a href="/wiki/Wikipedia:Citation_needed" title="Wikipedia:Citation needed"><span title="Each release takes time. More release should mean more time. How is time reduced as releases are increased? (February 2023)">citation needed</span></a></i>&#93;</sup> and risk of delivering changes by allowing for more incremental updates to applications in production. A straightforward and repeatable deployment process is important for continuous delivery.
</p>
<meta property="mw:PageProp/toc" />
<div class="mw-heading mw-heading2"><h2 id="Principles">Principles</h2><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Continuous_delivery&amp;action=edit&amp;section=1" title="Edit section: Principles"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></div>
<p>According to Neal Ford, continuous delivery adopts "Bring the pain forward," tackling tough tasks early, fostering automation and swift issue detection. <sup id="cite_ref-:0_3-0" class="reference"><a href="#cite_note-:0-3"><span class="cite-bracket">&#91;</span>3<span class="cite-bracket">&#93;</span></a></sup>
</p><p>Continuous delivery treats the commonplace notion of a <i>deployment pipeline</i><sup id="cite_ref-4" class="reference"><a href="#cite_note-4"><span class="cite-bracket">&#91;</span>4<span class="cite-bracket">&#93;</span></a></sup> as a <a href="/wiki/Lean_software_development" title="Lean software development">lean</a> <a href="/wiki/Poka-Yoke" class="mw-redirect" title="Poka-Yoke">Poka-Yoke</a>:<sup id="cite_ref-5" class="reference"><a href="#cite_note-5"><span class="cite-bracket">&#91;</span>5<span class="cite-bracket">&#93;</span></a></sup> a set of validations through which a piece of software must pass on its way to <a href="/wiki/Software_release" class="mw-redirect" title="Software release">release</a>. Code is compiled if necessary and then packaged by a build server every time a change is committed to a <a href="/wiki/Source_control" class="mw-redirect" title="Source control">source control repository</a>, then tested by a number of different techniques (possibly including manual testing) before it can be marked as releasable.
</p><p>Developers used to a long cycle time may need to change their mindset when working in a CD environment. Any code commit may be released to customers at any point. Patterns such as <a href="/wiki/Feature_toggle" title="Feature toggle">feature toggles</a> can be very useful for committing code early which is not yet ready for use by end users. Using <a href="/wiki/NoSQL" title="NoSQL">NoSQL</a> can eliminate the step of data migrations and schema changes, often manual steps or exceptions to a continuous delivery workflow.<sup id="cite_ref-6" class="reference"><a href="#cite_note-6"><span class="cite-bracket">&#91;</span>6<span class="cite-bracket">&#93;</span></a></sup> Other useful techniques for developing code in isolation such as <a href="/wiki/Branching_(software)" class="mw-redirect" title="Branching (software)">code branching</a> are not obsolete in a CD world, but must be adapted to fit the principles of CD - for example, running multiple long-lived code branches can prove impractical, as a releasable artifact must be built early in the CD process from a single code branch if it is to pass through all phases of the pipeline.<sup class="noprint Inline-Template" style="margin-left:0.1em; white-space:nowrap;">&#91;<i><a href="/wiki/Wikipedia:Please_clarify" title="Wikipedia:Please clarify"><span title="Unclear why making master branch releasable prevents other branches from being otherwise (September 2017)">clarification needed</span></a></i>&#93;</sup>
</p>
<div class="mw-heading mw-heading2"><h2 id="Deployment_pipeline">Deployment pipeline</h2><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Continuous_delivery&amp;action=edit&amp;section=2" title="Edit section: Deployment pipeline"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></div>
<figure class="mw-default-size" typeof="mw:File/Thumb"><a href="/wiki/File:Continuous_Delivery_process_diagram.svg" class="mw-file-description"><img src="//upload.wikimedia.org/wikipedia/commons/thumb/c/c3/Continuous_Delivery_process_diagram.svg/220px-Continuous_Delivery_process_diagram.svg.png" decoding="async" width="220" height="158" class="mw-file-element" srcset="//upload.wikimedia.org/wikipedia/commons/thumb/c/c3/Continuous_Delivery_process_diagram.svg/330px-Continuous_Delivery_process_diagram.svg.png 1.5x, //upload.wikimedia.org/wikipedia/commons/thumb/c/c3/Continuous_Delivery_process_diagram.svg/440px-Continuous_Delivery_process_diagram.svg.png 2x" data-file-width="780" data-file-height="559" /></a><figcaption></figcaption></figure>
<p>Continuous delivery is enabled through the deployment pipeline. The purpose of the deployment pipeline has three components: visibility, feedback, and continually deploy.<sup id="cite_ref-7" class="reference"><a href="#cite_note-7"><span class="cite-bracket">&#91;</span>7<span class="cite-bracket">&#93;</span></a></sup>
</p>
<ul><li><b>Visibility</b> – All aspects of the delivery system including building, deploying, testing, and releasing are visible to every member of the team to promote collaboration.</li>
<li><b>Feedback</b> – Team members learn of problems as soon as possible when they occur so that they are able to fix them as quickly as possible.</li>
<li><b>Continually deploy</b> – Through a fully automated process, you can deploy and release any version of the software to any environment.</li></ul>
<p><br />
According to Yan Cui, when it comes to <a href="/wiki/Serverless_computing" title="Serverless computing">serverless environments</a>, ephemeral resources should be kept together and have their own deployment pipeline to achieve a high <a href="/wiki/Cohesion_(computer_science)" title="Cohesion (computer science)">cohesion</a>. However, shared resources that have a long spin-up time and landing zone should have their own separate <a href="/wiki/Repository_(version_control)" title="Repository (version control)">repository</a>, <a href="/wiki/Deployment_pipeline" class="mw-redirect" title="Deployment pipeline">deployment pipeline</a> and stack. <sup id="cite_ref-:02_8-0" class="reference"><a href="#cite_note-:02-8"><span class="cite-bracket">&#91;</span>8<span class="cite-bracket">&#93;</span></a></sup>
</p>
<div class="mw-heading mw-heading2"><h2 id="Tools/tool_types"><span id="Tools.2Ftool_types"></span>Tools/tool types</h2><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Continuous_delivery&amp;action=edit&amp;section=3" title="Edit section: Tools/tool types"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></div>
<p>Continuous delivery takes automation from source control all the way through production. There are various tools that help accomplish all or part of this process.<sup id="cite_ref-9" class="reference"><a href="#cite_note-9"><span class="cite-bracket">&#91;</span>9<span class="cite-bracket">&#93;</span></a></sup> These tools are part of the deployment pipeline which includes continuous delivery. The types of tools that execute various parts of the process include: <a href="/wiki/Continuous_integration" title="Continuous integration">continuous integration</a>, <a href="/wiki/Application_release_automation" class="mw-redirect" title="Application release automation">application release automation</a>, <a href="/wiki/Build_automation" title="Build automation">build automation</a>, <a href="/wiki/Application_lifecycle_management" title="Application lifecycle management">application lifecycle management</a>.<sup id="cite_ref-10" class="reference"><a href="#cite_note-10"><span class="cite-bracket">&#91;</span>10<span class="cite-bracket">&#93;</span></a></sup>
</p>
<div class="mw-heading mw-heading2"><h2 id="Architecting_for_continuous_delivery">Architecting for continuous delivery</h2><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Continuous_delivery&amp;action=edit&amp;section=4" title="Edit section: Architecting for continuous delivery"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></div>
<p>To practice continuous delivery effectively, software applications have to meet a set of <a href="/wiki/Architecturally_significant_requirements" title="Architecturally significant requirements">architecturally significant requirements</a> (ASRs) such as deployability, modifiability, and testability.<sup id="cite_ref-ArchCD_LC_11-0" class="reference"><a href="#cite_note-ArchCD_LC-11"><span class="cite-bracket">&#91;</span>11<span class="cite-bracket">&#93;</span></a></sup> These ASRs require a high priority and cannot be traded off lightly.
</p><p><a href="/wiki/Microservices" title="Microservices">Microservices</a> are often used when architecting for continuous delivery.<sup id="cite_ref-Micro_Chen_12-0" class="reference"><a href="#cite_note-Micro_Chen-12"><span class="cite-bracket">&#91;</span>12<span class="cite-bracket">&#93;</span></a></sup> The use of Microservices can increase a software system's deployability and modifiability. The observed deployability improvements include: deployment independence, shorter deployment time, simpler deployment procedures, and zero downtime deployment. The observed modifiability improvements include: shorter cycle time for small incremental functional changes, easier technology selection changes, incremental quality attribute changes, and easier language and library upgrades.<sup id="cite_ref-Micro_Chen_12-1" class="reference"><a href="#cite_note-Micro_Chen-12"><span class="cite-bracket">&#91;</span>12<span class="cite-bracket">&#93;</span></a></sup>
</p>
<div class="mw-heading mw-heading2"><h2 id="Implementation_and_usage">Implementation and usage</h2><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Continuous_delivery&amp;action=edit&amp;section=5" title="Edit section: Implementation and usage"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></div>
<p>The original CD book written by <a href="/w/index.php?title=Jez_Humble&amp;action=edit&amp;redlink=1" class="new" title="Jez Humble (page does not exist)">Jez Humble</a> and David Farley (2010) popularized the term; however, since its creation the definition has continued to advance and now has a more developed meaning. Companies today are implementing these continuous delivery principles and best practices. The difference in domains, e.g. medical vs. web, is still significant and affects the implementation and usage.<sup id="cite_ref-:1_13-0" class="reference"><a href="#cite_note-:1-13"><span class="cite-bracket">&#91;</span>13<span class="cite-bracket">&#93;</span></a></sup> Well-known companies that have this approach include <a href="/wiki/Yahoo!" class="mw-redirect" title="Yahoo!">Yahoo!</a>,<sup id="cite_ref-14" class="reference"><a href="#cite_note-14"><span class="cite-bracket">&#91;</span>14<span class="cite-bracket">&#93;</span></a></sup> <a href="/wiki/Amazon_(company)" title="Amazon (company)">Amazon</a>,<sup id="cite_ref-15" class="reference"><a href="#cite_note-15"><span class="cite-bracket">&#91;</span>15<span class="cite-bracket">&#93;</span></a></sup> <a href="/wiki/Facebook" title="Facebook">Facebook</a>,<sup id="cite_ref-16" class="reference"><a href="#cite_note-16"><span class="cite-bracket">&#91;</span>16<span class="cite-bracket">&#93;</span></a></sup> <a href="/wiki/Google" title="Google">Google</a>,<sup id="cite_ref-17" class="reference"><a href="#cite_note-17"><span class="cite-bracket">&#91;</span>17<span class="cite-bracket">&#93;</span></a></sup> <a href="/wiki/Paddy_Power" title="Paddy Power">Paddy Power</a><sup id="cite_ref-CD_LC_1-1" class="reference"><a href="#cite_note-CD_LC-1"><span class="cite-bracket">&#91;</span>1<span class="cite-bracket">&#93;</span></a></sup> and <a href="/wiki/Wells_Fargo" title="Wells Fargo">Wells Fargo</a>.<sup id="cite_ref-18" class="reference"><a href="#cite_note-18"><span class="cite-bracket">&#91;</span>18<span class="cite-bracket">&#93;</span></a></sup>
</p>
<div class="mw-heading mw-heading2"><h2 id="Benefits_and_obstacles">Benefits and obstacles</h2><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Continuous_delivery&amp;action=edit&amp;section=6" title="Edit section: Benefits and obstacles"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></div>
<p>Several benefits of continuous delivery have been reported.<sup id="cite_ref-CD_LC_1-2" class="reference"><a href="#cite_note-CD_LC-1"><span class="cite-bracket">&#91;</span>1<span class="cite-bracket">&#93;</span></a></sup><sup id="cite_ref-:1_13-1" class="reference"><a href="#cite_note-:1-13"><span class="cite-bracket">&#91;</span>13<span class="cite-bracket">&#93;</span></a></sup>
</p>
<ul><li>Accelerated <a href="/wiki/Time_to_market" title="Time to market">time to market</a>: Continuous delivery lets an organization deliver the business value inherent in new software releases to customers more quickly. This capability helps the company stay a step ahead of the competition.</li>
<li>Building the right product: Frequent releases let the application development teams obtain user feedback more quickly. This lets them work on only the useful features. If they find that a feature isn't useful, they spend no further effort on it. This helps them build the right product.</li>
<li>Improved productivity and efficiency: Significant time savings for developers, testers, operations engineers, etc. through automation.</li>
<li>Reliable releases: The risks associated with a release have significantly decreased, and the release process has become more reliable. With continuous delivery, the deployment process and scripts are tested repeatedly before deployment to production. So, most errors in the deployment process and scripts have already been discovered. With more frequent releases, the number of code changes in each release decreases. This makes finding and fixing any problems that do occur easier, reducing the time in which they have an impact.</li>
<li>Improved <a href="/wiki/Product_quality" class="mw-redirect" title="Product quality">product quality</a>: The number of open bugs and production incidents has decreased significantly.</li>
<li>Improved <a href="/wiki/Customer_satisfaction" title="Customer satisfaction">customer satisfaction</a>: A higher level of customer satisfaction is achieved.</li></ul>
<p>Obstacles have also been investigated.<sup id="cite_ref-:1_13-2" class="reference"><a href="#cite_note-:1-13"><span class="cite-bracket">&#91;</span>13<span class="cite-bracket">&#93;</span></a></sup>
</p>
<ul><li>Customer preferences: Some customers do not want frequent updates to their systems.</li>
<li>Domain restrictions: In some domains, such as telecom, medical, avionics, railway and heavy industries, regulations require customer-side or even on-site testing of new versions.</li>
<li>Lack of test automation: Lack of test automation leads to a lack of developer confidence and can prevent using continuous delivery.</li>
<li>Differences in environments: Different environments used in the development, testing and production can result in undetected issues slipping to the production environment.</li>
<li>Tests needing a human <a href="/wiki/Test_oracle" title="Test oracle">oracle</a>: Not all quality attributes can be verified with automation. These attributes require humans in the loop, slowing down the delivery pipeline.</li></ul>
<p>Eight further adoption challenges were raised and elaborated on by Chen.<sup id="cite_ref-Overcome-challenges_19-0" class="reference"><a href="#cite_note-Overcome-challenges-19"><span class="cite-bracket">&#91;</span>19<span class="cite-bracket">&#93;</span></a></sup> These challenges are in the areas of organizational structure, processes, tools, infrastructure, legacy systems, architecting for continuous delivery, continuous testing of non-functional requirements, and test execution optimization.
</p>

<p>Several strategies to overcome continuous delivery adoption challenges have been reported.<sup id="cite_ref-Overcome-challenges_19-1" class="reference"><a href="#cite_note-Overcome-challenges-19"><span class="cite-bracket">&#91;</span>19<span class="cite-bracket">&#93;</span></a></sup>

</caption>
<tbody><tr>
<th>Strategy</th>
<th>Description
</th></tr>
<tr>
<td>Selling CD as a painkiller</td>
<td>Identify each stakeholder's pain points that CD can solve, and sell CD as a painkiller to that stakeholder. This strategy helps to achieve buy-in from the wide range of stakeholders that a CD implementation requires.
</td></tr>
<tr>
<td>Dedicated team with multi-disciplinary members</td>
<td>Without a dedicated team, it can be hard to progress because employees are often assigned to work on other value streams. A multi-disciplinary team not only provides the wide range of skills required for CD implementation but also smooths the communication with related teams.
</td></tr>
<tr>
<td>Continuous delivery of continuous delivery</td>
<td>Organize the implementation of CD in a way that delivers value to the company as early as possible, onboarding more projects gradually, in small increments and eventually rolling out CD across the whole organization. This strategy helps justify the investment required by making concrete benefits visible along the way. Visible benefits, in turn, help to achieve the sustained company support and investment required to survive the long and tough journey to CD.
</td></tr>
<tr>
<td>Starting with easy but important applications</td>
<td>When selecting the first few applications to migrate to CD, choose the ones that are easy to migrate but that are important to the business. Being easy to migrate helps to demonstrate the benefits of CD quickly, which can prevent the implementation initiative from being killed. Being important to the business helps to secure the required resources, demonstrates clear and unarguable value, and raises the visibility of CD in the organization.
</td></tr>
<tr>
<td>Visual CD pipeline skeleton</td>
<td>Give a team a visual CD pipeline skeleton that has the full CD pipeline view but with empty stages for those they cannot implement yet. This helps to build up a CD mindset and maintain the momentum for CD adoption. The pipeline skeleton is especially useful when the team's migration to CD requires a large effort and mindset changes over a long period of time.
</td></tr>
<tr>
<td>Expert drop</td>
<td>Assign a CD expert to join tough projects as a senior member of the development team. Having the expert on the team helps to build the motivation and momentum to move to CD from inside the team. It also helps to maintain momentum when the migration requires a large effort and a long period of time.
</td></tr></tbody></table>
<div class="mw-heading mw-heading2"><h2 id="Best_practices_for_cloud_systems">Best practices for cloud systems</h2><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Continuous_delivery&amp;action=edit&amp;section=8" title="Edit section: Best practices for cloud systems"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></div>
<p>The following practices can enhance productivity of <a href="/wiki/CI/CD" title="CI/CD">pipelines</a>, especially in systems hosted in the <a href="/wiki/Cloud_computing" title="Cloud computing">cloud</a>: <sup id="cite_ref-20" class="reference"><a href="#cite_note-20"><span class="cite-bracket">&#91;</span>20<span class="cite-bracket">&#93;</span></a></sup><sup id="cite_ref-21" class="reference"><a href="#cite_note-21"><span class="cite-bracket">&#91;</span>21<span class="cite-bracket">&#93;</span></a></sup><sup id="cite_ref-22" class="reference"><a href="#cite_note-22"><span class="cite-bracket">&#91;</span>22<span class="cite-bracket">&#93;</span></a></sup>
</p>
<ul><li><b>Number of Pipelines</b>: Small teams can be more productive by having one <a href="/wiki/Repository_(version_control)" title="Repository (version control)">repository</a> and one pipeline. In contrast, larger organizations may have separate repositories and pipelines for each team or even separate repositories and pipelines for each service within a team.</li>
<li><b>Permissions</b>: In the context of <a href="/wiki/CI/CD" title="CI/CD">pipeline-related</a> permissions, adhering to the <a href="/wiki/Principle_of_least_privilege" title="Principle of least privilege">principle of least privilege</a> can be challenging due to the dynamic nature of <a href="/wiki/Software_architecture" title="Software architecture">architecture</a>. Administrators may opt for more permissive permissions while implementing compensating <a href="/wiki/Security_controls" title="Security controls">security controls</a> to minimize the blast radius.</li></ul>
<div class="mw-heading mw-heading2"><h2 id="Relationship_to_DevOps">Relationship to DevOps</h2><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Continuous_delivery&amp;action=edit&amp;section=9" title="Edit section: Relationship to DevOps"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></div>
<style data-mw-deduplicate="TemplateStyles:r1236090951">.mw-parser-output .hatnote{font-style:italic}.mw-parser-output div.hatnote{padding-left:1.6em;margin-bottom:0.5em}.mw-parser-output .hatnote i{font-style:normal}.mw-parser-output .hatnote+link+.hatnote{margin-top:-0.5em}@media print{body.ns-0 .mw-parser-output .hatnote{display:none!important}}</style><div role="note" class="hatnote navigation-not-searchable">Main article: <a href="/wiki/DevOps" title="DevOps">DevOps</a></div>
<p><i>DevOps</i> is a software engineering approach that centers around cultural change, specifically the collaboration of the various teams involved in software delivery (developers, operations, quality assurance, management, etc.), as well as automating the processes in software delivery.<sup id="cite_ref-CD_HJ_23-0" class="reference"><a href="#cite_note-CD_HJ-23"><span class="cite-bracket">&#91;</span>23<span class="cite-bracket">&#93;</span></a></sup><sup id="cite_ref-24" class="reference"><a href="#cite_note-24"><span class="cite-bracket">&#91;</span>24<span class="cite-bracket">&#93;</span></a></sup><sup id="cite_ref-25" class="reference"><a href="#cite_note-25"><span class="cite-bracket">&#91;</span>25<span class="cite-bracket">&#93;</span></a></sup>

<div class="mw-heading mw-heading2"><h2 id="Relationship_to_Continuous_Deployment">Relationship to Continuous Deployment</h2><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Continuous_delivery&amp;action=edit&amp;section=10" title="Edit section: Relationship to Continuous Deployment"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></div>
<link rel="mw-deduplicated-inline-style" href="mw-data:TemplateStyles:r1236090951" /><div role="note" class="hatnote navigation-not-searchable">Main article: <a href="/wiki/Continuous_deployment" title="Continuous deployment">Continuous deployment</a></div>
<p><i>Continuous deployment</i> is a software engineering approach which uses automated software deployments.<sup id="cite_ref-Overcome-challenges_19-2" class="reference"><a href="#cite_note-Overcome-challenges-19"><span class="cite-bracket">&#91;</span>19<span class="cite-bracket">&#93;</span></a></sup>
In it, software is produced in short cycles but through automated <a href="/wiki/Software_deployment" title="Software deployment">software deployments</a> even to production rather than requiring a "click of a button" for that last step.<sup id="cite_ref-CD_LC_1-3" class="reference"><a href="#cite_note-CD_LC-1"><span class="cite-bracket">&#91;</span>1<span class="cite-bracket">&#93;</span></a></sup><sup class="reference nowrap">&#58;&#8202;<span title="Page: 52&#10;Quotation: &quot;Deployment into production takes just the click of a button&quot;" class="tooltip tooltip-dashed" style="border-bottom: 1px dashed;">52</span>&#8202;</sup> Therefore, continuous deployment can be considered a more sophisticated form of automation.<sup id="cite_ref-26" class="reference"><a href="#cite_note-26"><span class="cite-bracket">&#91;</span>26<span class="cite-bracket">&#93;</span></a></sup>
Academic literature differentiates between <i>continuous delivery</i> and <i>continuous deployment</i> according to deployment method; manual vs. automated.<sup id="cite_ref-shahin-et-al-2017_2-1" class="reference"><a href="#cite_note-shahin-et-al-2017-2"><span class="cite-bracket">&#91;</span>2<span class="cite-bracket">&#93;</span></a></sup><sup id="cite_ref-27" class="reference"><a href="#cite_note-27"><span class="cite-bracket">&#91;</span>27<span class="cite-bracket">&#93;</span></a></sup>
</p>
