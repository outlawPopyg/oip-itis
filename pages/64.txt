					<div id="mw-content-text" class="mw-body-content"><div class="mw-content-ltr mw-parser-output" lang="en" dir="ltr"><div class="shortdescription nomobile noexcerpt noprint searchaux" style="display:none">Architectural style for client-server applications</div>
<style data-mw-deduplicate="TemplateStyles:r1236090951">.mw-parser-output .hatnote{font-style:italic}.mw-parser-output div.hatnote{padding-left:1.6em;margin-bottom:0.5em}.mw-parser-output .hatnote i{font-style:normal}.mw-parser-output .hatnote+link+.hatnote{margin-top:-0.5em}@media print{body.ns-0 .mw-parser-output .hatnote{display:none!important}}</style><div role="note" class="hatnote navigation-not-searchable">For other uses, see <a href="/wiki/Rest_(disambiguation)" class="mw-redirect mw-disambig" title="Rest (disambiguation)">Rest (disambiguation)</a>.</div>
<p><b>REST</b> (<b>Re</b>presentational <b>S</b>tate <b>T</b>ransfer) is a <a href="/wiki/Software_architectural_style" class="mw-redirect" title="Software architectural style">software architectural style</a> that was created to describe the design and guide the development of the architecture for the <a href="/wiki/World_Wide_Web" title="World Wide Web">World Wide Web</a>. REST defines a set of constraints for how the architecture of a distributed, <a href="/wiki/Internet" title="Internet">Internet</a>-scale <a href="/wiki/Hypermedia" title="Hypermedia">hypermedia</a> system, such as the Web, should behave. The REST architectural style emphasises uniform <a href="/wiki/API" title="API">interfaces</a>, independent deployment of <a href="/wiki/Software_component" title="Software component">components</a>, the <a href="/wiki/Scalability" title="Scalability">scalability</a> of interactions between them, and creating a <a href="/wiki/Multitier_architecture" title="Multitier architecture">layered architecture</a> to promote <a href="/wiki/Caching" class="mw-redirect" title="Caching">caching</a> to reduce user-perceived <a href="/wiki/Latency_(engineering)" title="Latency (engineering)">latency</a>, enforce <a href="/wiki/Computer_security" title="Computer security">security</a>, and encapsulate <a href="/wiki/Legacy_system" title="Legacy system">legacy systems</a>.<sup id="cite_ref-Fielding-Ch5_1-0" class="reference"><a href="#cite_note-Fielding-Ch5-1"><span class="cite-bracket">&#91;</span>1<span class="cite-bracket">&#93;</span></a></sup>
</p><p>REST has been employed throughout the software industry to create <a href="/wiki/Stateless_protocol" title="Stateless protocol">stateless</a>, reliable <a href="/wiki/Web_application" title="Web application">web-based applications</a>.  An application that adheres to the <a href="#Architectural_constraints">REST architectural constraints</a> may be informally described as <i>RESTful</i>, although this term is more commonly associated with the design of <a href="/wiki/HTTP" title="HTTP">HTTP</a>-based <a href="/wiki/API" title="API">APIs</a> and what are widely considered best practices regarding the "verbs" (<a href="/wiki/Hypertext_Transfer_Protocol#Request_methods" class="mw-redirect" title="Hypertext Transfer Protocol">HTTP methods</a>) a <a href="/wiki/Web_resource" title="Web resource">resource</a> responds to, while having little to do with REST as originally formulated—and is often even at odds with the concept.<sup id="cite_ref-2" class="reference"><a href="#cite_note-2"><span class="cite-bracket">&#91;</span>2<span class="cite-bracket">&#93;</span></a></sup>
</p>
<meta property="mw:PageProp/toc" />
<div class="mw-heading mw-heading2"><h2 id="Principle">Principle</h2><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=REST&amp;action=edit&amp;section=1" title="Edit section: Principle"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></div>
<p>The term <i>representational state transfer</i> was introduced and defined in 2000 by computer scientist <a href="/wiki/Roy_Fielding" title="Roy Fielding">Roy Fielding</a> in his doctoral dissertation. It means that a server will respond with the representation of a resource (today, it will most often be an <a href="/wiki/HTML" title="HTML">HTML</a> document) and that resource will contain <a href="/wiki/Hypermedia" title="Hypermedia">hypermedia</a> links that can be followed to make the state of the system change. Any such request will in turn receive the representation of a resource, and so on.
</p><p>An important consequence is that the only identifier that needs to be known is the identifier of the first resource requested, and all other identifiers will be discovered. This means that those identifiers can change without the need to inform the client beforehand and that client and server must be inherently <a href="/wiki/Loose_coupling" title="Loose coupling">loosely coupled</a>.
</p>
<div class="mw-heading mw-heading2"><h2 id="History">History</h2><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=REST&amp;action=edit&amp;section=2" title="Edit section: History"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></div>
<figure class="mw-default-size" typeof="mw:File/Thumb"><a href="/wiki/File:Roy_Fielding_at_OSCON_2008.jpg" class="mw-file-description"><img src="//upload.wikimedia.org/wikipedia/commons/thumb/d/d9/Roy_Fielding_at_OSCON_2008.jpg/220px-Roy_Fielding_at_OSCON_2008.jpg" decoding="async" width="220" height="290" class="mw-file-element" srcset="//upload.wikimedia.org/wikipedia/commons/thumb/d/d9/Roy_Fielding_at_OSCON_2008.jpg/330px-Roy_Fielding_at_OSCON_2008.jpg 1.5x, //upload.wikimedia.org/wikipedia/commons/thumb/d/d9/Roy_Fielding_at_OSCON_2008.jpg/440px-Roy_Fielding_at_OSCON_2008.jpg 2x" data-file-width="2074" data-file-height="2734" /></a><figcaption><a href="/wiki/Roy_Fielding" title="Roy Fielding">Roy Fielding</a> speaking at <a href="/wiki/O%27Reilly_Open_Source_Convention" title="O&#39;Reilly Open Source Convention">OSCON</a> 2008</figcaption></figure>
<p>The Web began to enter everyday use in 1993–1994, when <a href="/wiki/List_of_websites_founded_before_1995" title="List of websites founded before 1995">websites for general use</a> started to become available.<sup id="cite_ref-3" class="reference"><a href="#cite_note-3"><span class="cite-bracket">&#91;</span>3<span class="cite-bracket">&#93;</span></a></sup> At the time, only a fragmented description existed of the Web's architecture, and there was pressure within the industry to agree on a standard for the Web interface protocols. For instance, several experimental extensions had been added to the communication protocol (HTTP) to support <a href="/wiki/Proxy_server" title="Proxy server">proxies</a>, and more extensions were being proposed, but there was a need for a formal Web architecture with which to evaluate the impact of these changes.<sup id="cite_ref-:0_4-0" class="reference"><a href="#cite_note-:0-4"><span class="cite-bracket">&#91;</span>4<span class="cite-bracket">&#93;</span></a></sup>
</p><p>The <a href="/wiki/W3C" class="mw-redirect" title="W3C">W3C</a> and <a href="/wiki/IETF" class="mw-redirect" title="IETF">IETF</a> <a href="/wiki/Working_group#Technical_working_groups" title="Working group">working groups</a> together started work on creating formal descriptions of the Web's three primary standards: <a href="/wiki/URI" class="mw-redirect" title="URI">URI</a>, <a href="/wiki/HTTP" title="HTTP">HTTP</a>, and <a href="/wiki/HTML" title="HTML">HTML</a>. Roy Fielding was involved in the creation of these standards (specifically HTTP&#160;1.0 and 1.1, and URI), and during the next six years he created the REST architectural style, testing its constraints on the Web's <a href="/wiki/Software_standard" title="Software standard"> protocol standards</a> and using it as a means to define architectural improvements&#160;— and to identify architectural mismatches. Fielding defined REST in his 2000 PhD dissertation "Architectural Styles and the Design of Network-based Software Architectures"<sup id="cite_ref-Fielding-Ch5_1-1" class="reference"><a href="#cite_note-Fielding-Ch5-1"><span class="cite-bracket">&#91;</span>1<span class="cite-bracket">&#93;</span></a></sup><sup id="cite_ref-5" class="reference"><a href="#cite_note-5"><span class="cite-bracket">&#91;</span>5<span class="cite-bracket">&#93;</span></a></sup> at <a href="/wiki/University_of_California,_Irvine" title="University of California, Irvine">UC Irvine</a>.
</p><p>To create the REST architectural style, Fielding identified the requirements that apply when creating a world-wide network-based application, such as the need for a low entry barrier to enable global adoption. He also surveyed many existing architectural styles for network-based applications, identifying which features are shared with other styles, such as caching and client–server features, and those which are unique to REST, such as the concept of resources. Fielding was trying to both categorise the existing architecture of the current implementation and identify which aspects should be considered central to the behavioural and performance requirements of the Web.
</p><p>By their nature, architectural styles are independent of any specific implementation, and while REST was created as part of the development of the Web standards, the implementation of the Web does not obey every constraint in the REST architectural style. Mismatches can occur due to ignorance or oversight, but the existence of the REST architectural style means that they can be identified before they become standardised. For example, Fielding identified the embedding of session information in URIs as a violation of the constraints of REST which can negatively affect shared caching and server scalability. <a href="/wiki/HTTP_cookies" class="mw-redirect" title="HTTP cookies">HTTP cookies</a> also violate REST constraints<sup id="cite_ref-:0_4-1" class="reference"><a href="#cite_note-:0-4"><span class="cite-bracket">&#91;</span>4<span class="cite-bracket">&#93;</span></a></sup> because they can become out of sync with the browser's application state, making them unreliable; they also contain opaque data that can be a concern for <a href="/wiki/Internet_privacy" title="Internet privacy">privacy</a> and <a href="/wiki/Internet_security" title="Internet security">security</a>.
</p>
<div class="mw-heading mw-heading2"><h2 id="Architectural_properties">Architectural properties</h2><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=REST&amp;action=edit&amp;section=3" title="Edit section: Architectural properties"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></div>
<p>The REST architectural style is designed for network-based applications, specifically client-server applications. But more than that, it is designed for Internet-scale usage, so the coupling between the <i>user agent</i> (client) and the <i>origin server</i> must be as <a href="/wiki/Loose_coupling" title="Loose coupling">loose</a> as possible to facilitate large-scale adoption.
</p><p>The strong decoupling of client and server together with the text-based transfer of information using a uniform addressing protocol provided the basis for meeting the requirements of the Web: <a href="/wiki/Extensibility" title="Extensibility">extensibility</a>, anarchic scalability<sup id="cite_ref-6" class="reference"><a href="#cite_note-6"><span class="cite-bracket">&#91;</span>6<span class="cite-bracket">&#93;</span></a></sup> and independent deployment of components, large-grain data transfer, and a low entry-barrier for content readers, content authors and developers.
</p>
<figure class="mw-halign-center" typeof="mw:File/Thumb"><a href="/wiki/File:REST_information_model.png" class="mw-file-description"><img src="//upload.wikimedia.org/wikipedia/commons/thumb/8/83/REST_information_model.png/900px-REST_information_model.png" decoding="async" width="900" height="355" class="mw-file-element" srcset="//upload.wikimedia.org/wikipedia/commons/thumb/8/83/REST_information_model.png/1350px-REST_information_model.png 1.5x, //upload.wikimedia.org/wikipedia/commons/thumb/8/83/REST_information_model.png/1800px-REST_information_model.png 2x" data-file-width="3963" data-file-height="1563" /></a><figcaption>An <a href="/wiki/Entity%E2%80%93relationship_model" title="Entity–relationship model">entity-relationship model</a> of the concepts expressed in the REST architectural style</figcaption></figure>
<p>The constraints of the REST architectural style affect the following architectural properties:<sup id="cite_ref-Fielding-Ch5_1-2" class="reference"><a href="#cite_note-Fielding-Ch5-1"><span class="cite-bracket">&#91;</span>1<span class="cite-bracket">&#93;</span></a></sup><sup id="cite_ref-SOA_with_REST_7-0" class="reference"><a href="#cite_note-SOA_with_REST-7"><span class="cite-bracket">&#91;</span>7<span class="cite-bracket">&#93;</span></a></sup>
</p>
<ul><li>Performance in component interactions, which can be the dominant factor in user-perceived performance and network efficiency;<sup id="cite_ref-Fielding-Ch2_8-0" class="reference"><a href="#cite_note-Fielding-Ch2-8"><span class="cite-bracket">&#91;</span>8<span class="cite-bracket">&#93;</span></a></sup></li>
<li><a href="/wiki/Scalability" title="Scalability">Scalability</a> allowing the support of large numbers of components and interactions among components;</li>
<li>Simplicity of a uniform interface;</li>
<li>Modifiability of components to meet changing needs (even while the application is running);</li>
<li>Visibility of communication between components by service agents;</li>
<li>Portability of components by moving program code with the data;</li>
<li>Reliability in the resistance to failure at the system level in the presence of failures within components, connectors, or data.<sup id="cite_ref-Fielding-Ch2_8-1" class="reference"><a href="#cite_note-Fielding-Ch2-8"><span class="cite-bracket">&#91;</span>8<span class="cite-bracket">&#93;</span></a></sup></li></ul>
<div class="mw-heading mw-heading2"><h2 id="Architectural_constraints">Architectural constraints</h2><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=REST&amp;action=edit&amp;section=4" title="Edit section: Architectural constraints"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></div>
<p>The REST architectural style defines six guiding constraints.<sup id="cite_ref-SOA_with_REST_7-1" class="reference"><a href="#cite_note-SOA_with_REST-7"><span class="cite-bracket">&#91;</span>7<span class="cite-bracket">&#93;</span></a></sup><sup id="cite_ref-Richardson_2007_9-0" class="reference"><a href="#cite_note-Richardson_2007-9"><span class="cite-bracket">&#91;</span>9<span class="cite-bracket">&#93;</span></a></sup> When these constraints are applied to the system architecture, it gains desirable <a href="/wiki/Non-functional_requirement" title="Non-functional requirement">non-functional properties</a>, such as performance, scalability, simplicity, modifiability, visibility, portability, and reliability.<sup id="cite_ref-Fielding-Ch5_1-3" class="reference"><a href="#cite_note-Fielding-Ch5-1"><span class="cite-bracket">&#91;</span>1<span class="cite-bracket">&#93;</span></a></sup>
</p><p>The formal REST constraints are as follows:<sup id="cite_ref-10" class="reference"><a href="#cite_note-10"><span class="cite-bracket">&#91;</span>10<span class="cite-bracket">&#93;</span></a></sup>
</p>
<ul><li>Client/Server – Clients are separated from servers by a well-defined interface</li>
<li>Stateless – A specific client does not consume server storage when the client is "at rest"</li>
<li>Cache – Responses indicate their own cacheability</li>
<li>Uniform interface</li>
<li>Layered system – A client cannot ordinarily tell whether it is connected directly to the end server, or to an intermediary along the way</li>
<li>Code on demand (optional) – Servers are able to temporarily extend or customize the functionality of a client by transferring logic to the client that can be executed within a standard virtual machine</li></ul>
<div class="mw-heading mw-heading3"><h3 id="Uniform_interface">Uniform interface</h3><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=REST&amp;action=edit&amp;section=5" title="Edit section: Uniform interface"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></div>
<p>The uniform interface constraint is fundamental to the design of any RESTful system.<sup id="cite_ref-Fielding-Ch5_1-4" class="reference"><a href="#cite_note-Fielding-Ch5-1"><span class="cite-bracket">&#91;</span>1<span class="cite-bracket">&#93;</span></a></sup> It simplifies and decouples the architecture, which enables each part to evolve independently. The four constraints for this uniform interface are:
</p>
<ul><li>Resource identification in requests: Individual resources are identified in requests using <a href="/wiki/Uniform_resource_identifier" class="mw-redirect" title="Uniform resource identifier">URIs</a>. The resources themselves are conceptually separate from the representations that are returned to the client. For example, the server could send data from its database as <a href="/wiki/HTML" title="HTML">HTML</a>, <a href="/wiki/XML" title="XML">XML</a> or as <a href="/wiki/JSON" title="JSON">JSON</a>—none of which are the server's internal representation.</li>
<li>Resource manipulation through representations: When a client holds a representation of a resource, including any <a href="/wiki/Metadata" title="Metadata">metadata</a> attached, it has enough information to modify or delete the resource's state.</li>
<li>Self-descriptive messages: Each message includes enough information to describe how to process the message. For example, which parser to invoke can be specified by a <a href="/wiki/Media_type" title="Media type">media type</a>.<sup id="cite_ref-Fielding-Ch5_1-5" class="reference"><a href="#cite_note-Fielding-Ch5-1"><span class="cite-bracket">&#91;</span>1<span class="cite-bracket">&#93;</span></a></sup></li>
<li>Hypermedia as the engine of application state (<a href="/wiki/HATEOAS" title="HATEOAS">HATEOAS</a>) – Having accessed an initial URI for the REST application—analogous to a human Web user accessing the <a href="/wiki/Home_page" title="Home page">home page</a> of a website—a REST client should then be able to use server-provided links dynamically to discover all the available resources it needs. As access proceeds, the server responds with text that includes <a href="/wiki/Hyperlink" title="Hyperlink">hyperlinks</a> to other resources that are currently available. There is no need for the client to be hard-coded with information regarding the structure of the server.<sup id="cite_ref-RESTfulAPI.net_11-0" class="reference"><a href="#cite_note-RESTfulAPI.net-11"><span class="cite-bracket">&#91;</span>11<span class="cite-bracket">&#93;</span></a></sup></li></ul>
