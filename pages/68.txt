					<div id="mw-content-text" class="mw-body-content"><div class="mw-content-ltr mw-parser-output" lang="en" dir="ltr"><div class="shortdescription nomobile noexcerpt noprint searchaux" style="display:none">Software design pattern</div>
<figure typeof="mw:File/Thumb"><a href="/wiki/File:Uluru,_helicopter_view,_cropped.jpg" class="mw-file-description"><img src="//upload.wikimedia.org/wikipedia/commons/thumb/c/c2/Uluru%2C_helicopter_view%2C_cropped.jpg/300px-Uluru%2C_helicopter_view%2C_cropped.jpg" decoding="async" width="300" height="197" class="mw-file-element" srcset="//upload.wikimedia.org/wikipedia/commons/thumb/c/c2/Uluru%2C_helicopter_view%2C_cropped.jpg/450px-Uluru%2C_helicopter_view%2C_cropped.jpg 1.5x, //upload.wikimedia.org/wikipedia/commons/thumb/c/c2/Uluru%2C_helicopter_view%2C_cropped.jpg/600px-Uluru%2C_helicopter_view%2C_cropped.jpg 2x" data-file-width="2876" data-file-height="1884" /></a><figcaption>Monolithic applications can be compared to <a href="/wiki/Monolith" title="Monolith">monoliths</a>, such as <a href="/wiki/Uluru" title="Uluru">Uluru</a>, <a href="/wiki/Australia" title="Australia">Australia</a>: a large single (mono) rock (lith)</figcaption></figure>
<p>In <a href="/wiki/Software_engineering" title="Software engineering">software engineering</a>, a <b>monolithic application</b> is a single unified <a href="/wiki/Application_software" title="Application software">software application</a> that is self-contained and independent from other applications, but typically lacks flexibility.<sup id="cite_ref-mishra_1-0" class="reference"><a href="#cite_note-mishra-1"><span class="cite-bracket">&#91;</span>1<span class="cite-bracket">&#93;</span></a></sup> There are advantages and disadvantages of building applications in a <a href="/wiki/Monolith" title="Monolith">monolithic</a> style of <a href="/wiki/Software_architecture" title="Software architecture">software architecture</a>, depending on requirements.<sup id="cite_ref-atlas_2-0" class="reference"><a href="#cite_note-atlas-2"><span class="cite-bracket">&#91;</span>2<span class="cite-bracket">&#93;</span></a></sup> Monolith applications are relatively simple and have a low cost but their shortcomings are lack of <a href="/wiki/Elasticity_(system_resource)" class="mw-redirect" title="Elasticity (system resource)">elasticity</a>, <a href="/wiki/Software_fault_tolerance" title="Software fault tolerance">fault tolerance</a> and <a href="/wiki/Scalability" title="Scalability">scalability</a>.<sup id="cite_ref-:0_3-0" class="reference"><a href="#cite_note-:0-3"><span class="cite-bracket">&#91;</span>3<span class="cite-bracket">&#93;</span></a></sup> Alternative styles to monolithic applications include <a href="/wiki/Multitier_architecture" title="Multitier architecture">multitier architectures</a>, <a href="/wiki/Distributed_computing" title="Distributed computing">distributed computing</a> and <a href="/wiki/Microservices" title="Microservices">microservices</a>.<sup id="cite_ref-atlas_2-1" class="reference"><a href="#cite_note-atlas-2"><span class="cite-bracket">&#91;</span>2<span class="cite-bracket">&#93;</span></a></sup> Despite their popularity in recent years, monolithic applications are still a good choice for applications with small team and little complexity. However, once it becomes too complex, you can consider refactoring it into microservices or a distributed application. Note that a monolithic application deployed on a single machine, may be performant enough for your current workload but it's less available, less durable, less changeable, less fine-tuned and less scalable than a well designed <a href="/wiki/Distributed_computing" title="Distributed computing">distributed system</a>.<sup id="cite_ref-4" class="reference"><a href="#cite_note-4"><span class="cite-bracket">&#91;</span>4<span class="cite-bracket">&#93;</span></a></sup>
</p><p>The <a href="/wiki/Design_philosophy" class="mw-redirect" title="Design philosophy">design philosophy</a> is that the application is responsible not just for a particular task, but can perform every step needed to complete a particular function.<sup id="cite_ref-5" class="reference"><a href="#cite_note-5"><span class="cite-bracket">&#91;</span>5<span class="cite-bracket">&#93;</span></a></sup> Some personal finance applications are monolithic in the sense that they help the user carry out a complete task, end to end, and are <a href="/wiki/Information_silo" title="Information silo">private data silos</a> rather than parts of a larger system of applications that work together. Some <a href="/wiki/Word_processor" title="Word processor">word processors</a> are monolithic applications.<sup id="cite_ref-6" class="reference"><a href="#cite_note-6"><span class="cite-bracket">&#91;</span>6<span class="cite-bracket">&#93;</span></a></sup> These applications are sometimes associated with <a href="/wiki/Mainframe_computers" class="mw-redirect" title="Mainframe computers">mainframe computers</a>.
</p><p>In software engineering, a monolithic application describes a software application that is designed as a single service.<sup id="cite_ref-7" class="reference"><a href="#cite_note-7"><span class="cite-bracket">&#91;</span>7<span class="cite-bracket">&#93;</span></a></sup> Multiple services can be desirable in certain scenarios as it can facilitate maintenance by allowing repair or replacement of parts of the application without requiring wholesale replacement.
</p><p>Modularity is achieved to various extents by different <a href="/wiki/Modular_programming" title="Modular programming">modular programming</a> approaches. Code-based modularity allows developers to reuse and repair parts of the application, but development tools are required to perform these maintenance functions (e.g. the application may need to be recompiled). Object-based modularity provides the application as a collection of separate executable files that may be independently maintained and replaced without redeploying the entire application (e.g. Microsoft's <a href="/wiki/Dynamic-link_library" title="Dynamic-link library">Dynamic-link library</a> (DLL); Sun/UNIX <a href="/wiki/Shared_object" class="mw-redirect" title="Shared object">shared object</a> files).<sup id="cite_ref-8" class="reference"><a href="#cite_note-8"><span class="cite-bracket">&#91;</span>8<span class="cite-bracket">&#93;</span></a></sup> Some object messaging capabilities allow object-based applications to be distributed across multiple computers (e.g. Microsoft's <a href="/wiki/Component_Object_Model" title="Component Object Model">Component Object Model</a> (COM)). <a href="/wiki/Service-oriented_architecture" title="Service-oriented architecture">Service-oriented architectures</a> use specific communication standards/protocols to communicate between modules.<sup class="noprint Inline-Template Template-Fact" style="white-space:nowrap;">&#91;<i><a href="/wiki/Wikipedia:Citation_needed" title="Wikipedia:Citation needed"><span title="This claim needs references to reliable sources. (May 2023)">citation needed</span></a></i>&#93;</sup>
</p><p>In its original use, the term "monolithic" described enormous mainframe applications with no usable modularity.<sup id="cite_ref-raymond_9-0" class="reference"><a href="#cite_note-raymond-9"><span class="cite-bracket">&#91;</span>9<span class="cite-bracket">&#93;</span></a></sup> This, in combination with the rapid increase in computational power and therefore rapid increase in the complexity of the problems which could be tackled by software, resulted in unmaintainable systems and the "<a href="/wiki/Software_crisis" title="Software crisis">software crisis</a>".
</p>
