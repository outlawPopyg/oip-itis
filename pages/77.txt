<p><b>Ansible</b> is a suite of software tools that enables <a href="/wiki/Infrastructure_as_code" title="Infrastructure as code">infrastructure as code</a>. It is <a href="/wiki/Open-source_software" title="Open-source software">open-source</a> and the suite includes software <a href="/wiki/Provisioning_(technology)" title="Provisioning (technology)">provisioning</a>, <a href="/wiki/Software_configuration_management" title="Software configuration management">configuration management</a>, and <a href="/wiki/Application_deployment" class="mw-redirect" title="Application deployment">application deployment</a> functionality.<sup id="cite_ref-2" class="reference"><a href="#cite_note-2"><span class="cite-bracket">&#91;</span>2<span class="cite-bracket">&#93;</span></a></sup>
</p><p>Originally written by Michael DeHaan in 2012, and acquired by <a href="/wiki/Red_Hat" title="Red Hat">Red Hat</a> in 2015, Ansible is designed to configure both <a href="/wiki/Unix-like" title="Unix-like">Unix-like</a> systems and <a href="/wiki/Microsoft_Windows" title="Microsoft Windows">Microsoft Windows</a>. Ansible is <a href="/wiki/Agentless" class="mw-redirect" title="Agentless">agentless</a>, relying on temporary remote connections via <a href="/wiki/SSH" class="mw-redirect" title="SSH">SSH</a> or <a href="/wiki/Windows_Remote_Management" title="Windows Remote Management">Windows Remote Management</a> which allows <a href="/wiki/PowerShell" title="PowerShell">PowerShell</a> execution. The Ansible control node runs on most Unix-like systems that are able to run <a href="/wiki/Python_(programming_language)" title="Python (programming language)">Python</a>, including Windows with <a href="/wiki/Windows_Subsystem_for_Linux" title="Windows Subsystem for Linux">Windows Subsystem for Linux</a> installed.<sup id="cite_ref-3" class="reference"><a href="#cite_note-3"><span class="cite-bracket">&#91;</span>3<span class="cite-bracket">&#93;</span></a></sup> <a href="/wiki/System_configuration" title="System configuration">System configuration</a> is defined in part by using its own <a href="/wiki/Declarative_language" class="mw-redirect" title="Declarative language">declarative language</a>.
</p>
<meta property="mw:PageProp/toc" />
<div class="mw-heading mw-heading2"><h2 id="History">History</h2><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Ansible_(software)&amp;action=edit&amp;section=1" title="Edit section: History"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></div>
<p>The term "<a href="/wiki/Ansible" title="Ansible">ansible</a>" was coined by <a href="/wiki/Ursula_K._Le_Guin" title="Ursula K. Le Guin">Ursula K. Le Guin</a> in her 1966 novel <i><a href="/wiki/Rocannon%27s_World" title="Rocannon&#39;s World">Rocannon's World</a></i>,<sup id="cite_ref-Bernardo_2006_p._18_4-0" class="reference"><a href="#cite_note-Bernardo_2006_p._18-4"><span class="cite-bracket">&#91;</span>4<span class="cite-bracket">&#93;</span></a></sup> and refers to fictional instantaneous communication systems.<sup id="cite_ref-FAQ_5-0" class="reference"><a href="#cite_note-FAQ-5"><span class="cite-bracket">&#91;</span>5<span class="cite-bracket">&#93;</span></a></sup><sup id="cite_ref-6" class="reference"><a href="#cite_note-6"><span class="cite-bracket">&#91;</span>6<span class="cite-bracket">&#93;</span></a></sup>
</p><p>The Ansible tool was developed by Michael DeHaan, the author of the provisioning server application <a href="/wiki/Cobbler_(software)" title="Cobbler (software)">Cobbler</a> and co-author of the <a href="/w/index.php?title=Fedora_Unified_Network_Controller&amp;action=edit&amp;redlink=1" class="new" title="Fedora Unified Network Controller (page does not exist)">Fedora Unified Network Controller</a> (Func) framework for remote administration.<sup id="cite_ref-7" class="reference"><a href="#cite_note-7"><span class="cite-bracket">&#91;</span>7<span class="cite-bracket">&#93;</span></a></sup>
</p><p>Ansible, Inc. (originally AnsibleWorks, Inc.) was the company founded in 2013 by DeHaan, Timothy Gerla, and Saïd Ziouani to commercially support and sponsor Ansible.<sup id="cite_ref-8" class="reference"><a href="#cite_note-8"><span class="cite-bracket">&#91;</span>8<span class="cite-bracket">&#93;</span></a></sup><sup id="cite_ref-9" class="reference"><a href="#cite_note-9"><span class="cite-bracket">&#91;</span>9<span class="cite-bracket">&#93;</span></a></sup><sup id="cite_ref-10" class="reference"><a href="#cite_note-10"><span class="cite-bracket">&#91;</span>10<span class="cite-bracket">&#93;</span></a></sup> <a href="/wiki/Red_Hat" title="Red Hat">Red Hat</a> acquired Ansible in October 2015.<sup id="cite_ref-11" class="reference"><a href="#cite_note-11"><span class="cite-bracket">&#91;</span>11<span class="cite-bracket">&#93;</span></a></sup><sup id="cite_ref-12" class="reference"><a href="#cite_note-12"><span class="cite-bracket">&#91;</span>12<span class="cite-bracket">&#93;</span></a></sup>
</p><p>Ansible is included as part of the <a href="/wiki/Fedora_(operating_system)" class="mw-redirect" title="Fedora (operating system)">Fedora</a> distribution of Linux, owned by <a href="/wiki/Red_Hat" title="Red Hat">Red Hat</a>, and is also available for <a href="/wiki/Red_Hat_Enterprise_Linux" title="Red Hat Enterprise Linux">Red Hat Enterprise Linux</a>, <a href="/wiki/CentOS" title="CentOS">CentOS</a>, <a href="/wiki/OpenSUSE" title="OpenSUSE">openSUSE</a>, <a href="/wiki/SUSE_Linux_Enterprise" title="SUSE Linux Enterprise">SUSE Linux Enterprise</a>, <a href="/wiki/Debian" title="Debian">Debian</a>, <a href="/wiki/Ubuntu" title="Ubuntu">Ubuntu</a>, <a href="/wiki/Scientific_Linux" title="Scientific Linux">Scientific Linux</a>, and <a href="/wiki/Oracle_Linux" title="Oracle Linux">Oracle Linux</a> via Extra Packages for Enterprise Linux,  as well as for other operating systems.<sup id="cite_ref-Ansible_Linux_packages_13-0" class="reference"><a href="#cite_note-Ansible_Linux_packages-13"><span class="cite-bracket">&#91;</span>13<span class="cite-bracket">&#93;</span></a></sup>
</p>
<div class="mw-heading mw-heading2"><h2 id="Architecture">Architecture</h2><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Ansible_(software)&amp;action=edit&amp;section=2" title="Edit section: Architecture"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></div>
<div class="mw-heading mw-heading3"><h3 id="Overview">Overview</h3><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Ansible_(software)&amp;action=edit&amp;section=3" title="Edit section: Overview"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></div>
<p>Ansible helps to manage multiple machines by selecting portions of Ansible's <a href="#Inventory_configuration">inventory</a> stored in simple <a href="/wiki/Plain_text" title="Plain text">plain text</a> files. The inventory is configurable, and target machine inventory can be sourced dynamically or from cloud-based sources in different formats (<a href="/wiki/YAML" title="YAML">YAML</a>, <a href="/wiki/INI_file" title="INI file">INI</a>).<sup id="cite_ref-Inventory_14-0" class="reference"><a href="#cite_note-Inventory-14"><span class="cite-bracket">&#91;</span>14<span class="cite-bracket">&#93;</span></a></sup>
</p><p>Sensitive data can be stored in encrypted files using Ansible Vault<sup id="cite_ref-15" class="reference"><a href="#cite_note-15"><span class="cite-bracket">&#91;</span>15<span class="cite-bracket">&#93;</span></a></sup> since 2014.<sup id="cite_ref-16" class="reference"><a href="#cite_note-16"><span class="cite-bracket">&#91;</span>16<span class="cite-bracket">&#93;</span></a></sup>
In contrast with other popular configuration-management software — such as <a href="/wiki/Chef_(software)" class="mw-redirect" title="Chef (software)">Chef</a>, <a href="/wiki/Puppet_(software)" title="Puppet (software)">Puppet</a>, <a href="/wiki/Salt_(software)" title="Salt (software)">Salt</a> and <a href="/wiki/CFEngine" title="CFEngine">CFEngine</a> — Ansible uses an <i>agentless</i> architecture,<sup id="cite_ref-The_Benefits_of_Agentless_Architecture_17-0" class="reference"><a href="#cite_note-The_Benefits_of_Agentless_Architecture-17"><span class="cite-bracket">&#91;</span>17<span class="cite-bracket">&#93;</span></a></sup> with
Ansible software not normally running or even installed on the controlled node.<sup id="cite_ref-The_Benefits_of_Agentless_Architecture_17-1" class="reference"><a href="#cite_note-The_Benefits_of_Agentless_Architecture-17"><span class="cite-bracket">&#91;</span>17<span class="cite-bracket">&#93;</span></a></sup>  Instead, Ansible orchestrates a node by installing and running <i>modules</i> on the node temporarily via SSH.  For the duration of an orchestration task, a process running the module communicates with the controlling machine with a <a href="/wiki/JSON" title="JSON">JSON</a>-based protocol via its standard input and output.<sup id="cite_ref-Ansible_in_Depth_18-0" class="reference"><a href="#cite_note-Ansible_in_Depth-18"><span class="cite-bracket">&#91;</span>18<span class="cite-bracket">&#93;</span></a></sup> When Ansible is not managing a node, it does not consume resources on the node because no <a href="/wiki/Daemon_(computing)" title="Daemon (computing)">daemons</a> are run or software installed.<sup id="cite_ref-The_Benefits_of_Agentless_Architecture_17-2" class="reference"><a href="#cite_note-The_Benefits_of_Agentless_Architecture-17"><span class="cite-bracket">&#91;</span>17<span class="cite-bracket">&#93;</span></a></sup>
</p>
<div class="mw-heading mw-heading4"><h4 id="Dependencies">Dependencies</h4><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Ansible_(software)&amp;action=edit&amp;section=4" title="Edit section: Dependencies"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></div>
<p>Ansible requires <a href="/wiki/Python_(programming_language)" title="Python (programming language)">Python</a> to be installed on all managing machines, including <a href="/wiki/Pip_(package_manager)" title="Pip (package manager)">pip</a> package manager along with configuration-management software and its dependent packages. Managed network devices require no extra dependencies and are agentless.<sup id="cite_ref-:1_19-0" class="reference"><a href="#cite_note-:1-19"><span class="cite-bracket">&#91;</span>19<span class="cite-bracket">&#93;</span></a></sup>
</p>
<div class="mw-heading mw-heading3"><h3 id="Control_node">Control node</h3><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Ansible_(software)&amp;action=edit&amp;section=5" title="Edit section: Control node"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></div>
<p>The control node (master host) is intended to manage (orchestrate) target machines (nodes termed as "<a href="#Inventory_configuration">inventory</a>", see below).<sup id="cite_ref-:2_20-0" class="reference"><a href="#cite_note-:2-20"><span class="cite-bracket">&#91;</span>20<span class="cite-bracket">&#93;</span></a></sup> Control nodes can be run from <a href="/wiki/Linux" title="Linux">Linux</a> and <a href="/wiki/Unix-like" title="Unix-like">Unix-like</a> operating systems (including <a href="/wiki/MacOS" title="MacOS">MacOS</a>); <a href="/wiki/Microsoft_Windows" title="Microsoft Windows">Windows</a> OSs are only supported through the <a href="/wiki/Windows_Subsystem_for_Linux" title="Windows Subsystem for Linux">Windows Subsystem for Linux</a>.<sup id="cite_ref-21" class="reference"><a href="#cite_note-21"><span class="cite-bracket">&#91;</span>21<span class="cite-bracket">&#93;</span></a></sup> Multiple control nodes are allowed.<sup id="cite_ref-:2_20-1" class="reference"><a href="#cite_note-:2-20"><span class="cite-bracket">&#91;</span>20<span class="cite-bracket">&#93;</span></a></sup> Ansible does not require a single controlling machine for orchestration,<sup id="cite_ref-:0_22-0" class="reference"><a href="#cite_note-:0-22"><span class="cite-bracket">&#91;</span>22<span class="cite-bracket">&#93;</span></a></sup> ensuring that disaster recovery is simple.<sup id="cite_ref-:0_22-1" class="reference"><a href="#cite_note-:0-22"><span class="cite-bracket">&#91;</span>22<span class="cite-bracket">&#93;</span></a></sup> Nodes are managed by the controlling node over <a href="/wiki/SSH" class="mw-redirect" title="SSH">SSH</a>.
</p>
<div class="mw-heading mw-heading3"><h3 id="Design_goals">Design goals</h3><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Ansible_(software)&amp;action=edit&amp;section=6" title="Edit section: Design goals"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></div>
<p>The design goals of Ansible include:<sup id="cite_ref-Ansible_in_Depth_18-1" class="reference"><a href="#cite_note-Ansible_in_Depth-18"><span class="cite-bracket">&#91;</span>18<span class="cite-bracket">&#93;</span></a></sup>
</p>
<ul><li>Minimal in nature. Management systems should not impose additional dependencies on the environment.<sup id="cite_ref-The_Benefits_of_Agentless_Architecture_17-3" class="reference"><a href="#cite_note-The_Benefits_of_Agentless_Architecture-17"><span class="cite-bracket">&#91;</span>17<span class="cite-bracket">&#93;</span></a></sup></li>
<li>Consistent. With Ansible, one should be able to create consistent environments.</li>
<li>Secure. Ansible does not deploy agents to nodes. Only <a href="/wiki/OpenSSH" title="OpenSSH">OpenSSH</a> and Python are required on the managed nodes.<sup id="cite_ref-The_Benefits_of_Agentless_Architecture_17-4" class="reference"><a href="#cite_note-The_Benefits_of_Agentless_Architecture-17"><span class="cite-bracket">&#91;</span>17<span class="cite-bracket">&#93;</span></a></sup><sup id="cite_ref-:0_22-2" class="reference"><a href="#cite_note-:0-22"><span class="cite-bracket">&#91;</span>22<span class="cite-bracket">&#93;</span></a></sup></li>
<li>Reliable. When carefully written, an Ansible <a href="#Playbooks">playbook</a> can be <a href="/wiki/Idempotent" class="mw-redirect" title="Idempotent">idempotent</a>, to prevent unexpected side effects on the managed systems.<sup id="cite_ref-Achieving_Rolling_Updates_and_Continuous_Deployment_with_Zero_Downtime_23-0" class="reference"><a href="#cite_note-Achieving_Rolling_Updates_and_Continuous_Deployment_with_Zero_Downtime-23"><span class="cite-bracket">&#91;</span>23<span class="cite-bracket">&#93;</span></a></sup> It is possible to write playbooks that are not idempotent.</li>
<li>Minimal learning required. Playbooks use an easy and descriptive language based on YAML and <a href="/wiki/Jinja_(template_engine)" title="Jinja (template engine)">Jinja templates</a>.</li></ul>
<div class="mw-heading mw-heading3"><h3 id="Modules">Modules</h3><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Ansible_(software)&amp;action=edit&amp;section=7" title="Edit section: Modules"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></div>
<p>Modules<sup id="cite_ref-24" class="reference"><a href="#cite_note-24"><span class="cite-bracket">&#91;</span>24<span class="cite-bracket">&#93;</span></a></sup> are mostly standalone and can be written in a standard scripting language (such as Python, <a href="/wiki/Perl" title="Perl">Perl</a>, <a href="/wiki/Ruby_(programming_language)" title="Ruby (programming language)">Ruby</a>, Bash, etc.)<sup class="noprint Inline-Template Template-Fact" style="white-space:nowrap;">&#91;<i><a href="/wiki/Wikipedia:Citation_needed" title="Wikipedia:Citation needed"><span title="I have been working on Ansible for about half a decade, and I know for a fact that you can write Ansible modules in any language, provided the language is Python. (September 2023)">citation needed</span></a></i>&#93;</sup>. One of the guiding goals of modules is idempotency, which means that even if an operation is repeated multiple times (e.g., upon recovery from an outage), it will always place the system into the same state.<sup id="cite_ref-Ansible_in_Depth_18-2" class="reference"><a href="#cite_note-Ansible_in_Depth-18"><span class="cite-bracket">&#91;</span>18<span class="cite-bracket">&#93;</span></a></sup><sup class="noprint Inline-Template noprint Template-Fact" style="white-space:nowrap;">&#91;<i><a href="/wiki/Wikipedia:No_original_research#Primary,_secondary_and_tertiary_sources" title="Wikipedia:No original research"><span title="This claim needs references to reliable secondary sources. (November 2021)">non-primary source needed</span></a></i>&#93;</sup>
</p>
<div class="mw-heading mw-heading3"><h3 id="Inventory_configuration">Inventory configuration</h3><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Ansible_(software)&amp;action=edit&amp;section=8" title="Edit section: Inventory configuration"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></div>
<p>Location of target nodes is specified through inventory configuration lists (INI or YAML formatted) located at <code class="mw-highlight mw-highlight-lang-text mw-content-ltr" style="" dir="ltr">/etc/ansible/hosts</code> (on Linux).<sup id="cite_ref-Inventory_14-1" class="reference"><a href="#cite_note-Inventory-14"><span class="cite-bracket">&#91;</span>14<span class="cite-bracket">&#93;</span></a></sup><sup id="cite_ref-25" class="reference"><a href="#cite_note-25"><span class="cite-bracket">&#91;</span>25<span class="cite-bracket">&#93;</span></a></sup> The configuration file lists either the <a href="/wiki/IP_address" title="IP address">IP address</a> or <a href="/wiki/Hostname" title="Hostname">hostname</a> of each node that is accessible by Ansible. In addition, nodes can be assigned to groups.<sup id="cite_ref-Inventory_14-2" class="reference"><a href="#cite_note-Inventory-14"><span class="cite-bracket">&#91;</span>14<span class="cite-bracket">&#93;</span></a></sup>
</p><p>An example inventory format (<a href="/wiki/INI_file" title="INI file">INI file</a>):
</p>
<div class="mw-highlight mw-highlight-lang-ini mw-content-ltr" dir="ltr"><pre><span></span><span class="na">192.168.6.1</span>

<span class="k">[webservers]</span>
<span class="na">foo.example.com</span>
<span class="na">bar.example.com</span>
</pre></div>
<p>This configuration file specifies three nodes: the first node is specified by an IP address, and the latter two nodes are specified by hostnames. Additionally, the latter two nodes are grouped under the <code>webservers</code> group.
</p><p>Ansible can also use a custom <i>Dynamic Inventory</i> script, which can dynamically pull data from a different system,<sup id="cite_ref-26" class="reference"><a href="#cite_note-26"><span class="cite-bracket">&#91;</span>26<span class="cite-bracket">&#93;</span></a></sup> and supports groups of groups.<sup id="cite_ref-27" class="reference"><a href="#cite_note-27"><span class="cite-bracket">&#91;</span>27<span class="cite-bracket">&#93;</span></a></sup>
</p>
<div class="mw-heading mw-heading3"><h3 id="Playbooks">Playbooks</h3><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Ansible_(software)&amp;action=edit&amp;section=9" title="Edit section: Playbooks"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></div>
<p>Playbooks are <a href="/wiki/YAML" title="YAML">YAML</a> files that store lists of tasks for repeated<sup id="cite_ref-Playbooks_28-0" class="reference"><a href="#cite_note-Playbooks-28"><span class="cite-bracket">&#91;</span>28<span class="cite-bracket">&#93;</span></a></sup><sup id="cite_ref-:2_20-2" class="reference"><a href="#cite_note-:2-20"><span class="cite-bracket">&#91;</span>20<span class="cite-bracket">&#93;</span></a></sup> executions on managed nodes.<sup id="cite_ref-:2_20-3" class="reference"><a href="#cite_note-:2-20"><span class="cite-bracket">&#91;</span>20<span class="cite-bracket">&#93;</span></a></sup><sup id="cite_ref-29" class="reference"><a href="#cite_note-29"><span class="cite-bracket">&#91;</span>29<span class="cite-bracket">&#93;</span></a></sup> Each Playbook maps (associates) a group of hosts to a set of roles. Each role is represented by calls to Ansible tasks.<sup id="cite_ref-30" class="reference"><a href="#cite_note-30"><span class="cite-bracket">&#91;</span>30<span class="cite-bracket">&#93;</span></a></sup>
</p>
<div class="mw-heading mw-heading3"><h3 id="Ansible_Automation_Platform">Ansible Automation Platform</h3><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Ansible_(software)&amp;action=edit&amp;section=10" title="Edit section: Ansible Automation Platform"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></div>
<p><span class="anchor" id="AWX"></span>
The Ansible Automation Platform (AAP) is a <a href="/wiki/REST_API" class="mw-redirect" title="REST API">REST API</a>, <a href="/wiki/Web_service" title="Web service">web service</a>, and <a href="/wiki/Web_application" title="Web application">web-based interface</a> (application) designed to make Ansible more accessible to people with a wide range of IT skillsets. It is a platform composed of multiple components including developer tooling, an operations interface, as well as an Automation Mesh to enable automation tasks at scale across data centers. AAP is a commercial product supported by Red Hat, Inc. but derived 17+ <a href="/wiki/Upstream_(software_development)" title="Upstream (software development)">upstream</a> open source projects including the AWX upstream project (formerly Ansible Tower derived from), which has been open source since September 2017.<sup id="cite_ref-31" class="reference"><a href="#cite_note-31"><span class="cite-bracket">&#91;</span>31<span class="cite-bracket">&#93;</span></a></sup><sup id="cite_ref-32" class="reference"><a href="#cite_note-32"><span class="cite-bracket">&#91;</span>32<span class="cite-bracket">&#93;</span></a></sup><sup id="cite_ref-33" class="reference"><a href="#cite_note-33"><span class="cite-bracket">&#91;</span>33<span class="cite-bracket">&#93;</span></a></sup><sup id="cite_ref-34" class="reference"><a href="#cite_note-34"><span class="cite-bracket">&#91;</span>34<span class="cite-bracket">&#93;</span></a></sup>
</p><p>There also is another open source alternative to Tower, Semaphore, written in <a href="/wiki/Go_(programming_language)" title="Go (programming language)">Go</a>.<sup id="cite_ref-35" class="reference"><a href="#cite_note-35"><span class="cite-bracket">&#91;</span>35<span class="cite-bracket">&#93;</span></a></sup><sup id="cite_ref-36" class="reference"><a href="#cite_note-36"><span class="cite-bracket">&#91;</span>36<span class="cite-bracket">&#93;</span></a></sup>
</p>
<div class="mw-heading mw-heading2"><h2 id="Platform_support">Platform support</h2><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Ansible_(software)&amp;action=edit&amp;section=11" title="Edit section: Platform support"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></div>
<p>Control machines have to be a Linux/Unix host (for example <a href="/wiki/BSD" class="mw-redirect" title="BSD">BSD</a>, <a href="/wiki/CentOS" title="CentOS">CentOS</a>, <a href="/wiki/Debian" title="Debian">Debian</a>, <a href="/wiki/MacOS" title="MacOS">macOS</a>, <a href="/wiki/Red_Hat_Enterprise_Linux" title="Red Hat Enterprise Linux">Red Hat Enterprise Linux</a>, <a href="/wiki/SUSE_Linux_Enterprise" title="SUSE Linux Enterprise">SUSE Linux Enterprise</a>, <a href="/wiki/Ubuntu" title="Ubuntu">Ubuntu</a><sup id="cite_ref-Ansible_Linux_packages_13-1" class="reference"><a href="#cite_note-Ansible_Linux_packages-13"><span class="cite-bracket">&#91;</span>13<span class="cite-bracket">&#93;</span></a></sup>), and Python 2.7 or 3.5 is required.<sup id="cite_ref-:1_19-1" class="reference"><a href="#cite_note-:1-19"><span class="cite-bracket">&#91;</span>19<span class="cite-bracket">&#93;</span></a></sup>
</p><p>Managed nodes, if they are Unix-like, must have Python 2.4 or later. For managed nodes with Python 2.5 or earlier, the <code>python-simplejson</code> package is also required.<sup id="cite_ref-Getting_Started_37-0" class="reference"><a href="#cite_note-Getting_Started-37"><span class="cite-bracket">&#91;</span>37<span class="cite-bracket">&#93;</span></a></sup> Since version 1.7, Ansible can also manage Windows<sup id="cite_ref-38" class="reference"><a href="#cite_note-38"><span class="cite-bracket">&#91;</span>38<span class="cite-bracket">&#93;</span></a></sup> nodes.<sup id="cite_ref-Getting_Started_37-1" class="reference"><a href="#cite_note-Getting_Started-37"><span class="cite-bracket">&#91;</span>37<span class="cite-bracket">&#93;</span></a></sup> In this case, native PowerShell remoting supported by the WS-Management protocol is used instead of SSH.
</p><p>Ansible can deploy to <a href="/wiki/Bare_metal" class="mw-redirect" title="Bare metal">bare metal</a> hosts, <a href="/wiki/Virtual_machines" class="mw-redirect" title="Virtual machines">virtual machines</a>, and cloud environments.<sup id="cite_ref-Ansible_in_Depth_18-3" class="reference"><a href="#cite_note-Ansible_in_Depth-18"><span class="cite-bracket">&#91;</span>18<span class="cite-bracket">&#93;</span></a></sup>
</p>
